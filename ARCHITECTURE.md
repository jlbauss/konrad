## The big picture

Konrad runs [opencode](https://github.com/sst/opencode) inside a sandboxed, rootless Podman container.

- **The image is the canonical artifact.** `image/Dockerfile` builds it; `bin/konrad` — a thin host-side bash CLI — is the primary consumer. A second consumption path — a `konrad init-devcontainer` that scaffolds a Dev Container into the user's own project — is on the [ROADMAP.md](ROADMAP.md).
- **Podman, not Docker.** Open-source, free for commercial use, ergonomic on macOS. `--userns=keep-id` shares the container `node` user's UID with the host user, so bind-mounted files have sane ownership. Docker support is on the roadmap.
- **Two engines, one abstraction.** Podman is the default everywhere. On Apple-Silicon macOS 26+ with Apple's [`container`](https://github.com/apple/container) CLI present, Konrad uses that native engine instead — dropping the `podman machine` VM. The two CLIs aren't subcommand-compatible, so every host-side engine call in `bin/konrad` goes through an `eng_*` wrapper and `detect_engine()` fixes the choice once at startup (`KONRAD_ENGINE` overrides). apple/container needs no `--userns` (VirtioFS maps guest UIDs → the host user) and rejects the `:z`/`:U`/`:ro` mount suffixes, so those are dropped there. Its named volumes are also root-owned and can't be chowned (no `:U`), so the three persistent stores (secrets/cache/state) are backed by host directories under `~/.local/state/konrad/store/` instead — VirtioFS remaps those to the container user, the same way it makes the `/workspace` bind writable. The dev container is Linux and always resolves to Podman.
- **Tuned for Qwen3.6-class local models.** The agent prompt ([image/opencode/agents/konrad.md](image/opencode/agents/konrad.md)) is sized for a ~30B-class local model; the tested recommendation is `qwen/qwen3.6-27b`. Frontier models work fine; sub-10B ones may need prompt softening.

## Engineering ethos — lightweight, composed, low-maintenance

The guiding principle behind every other choice here: **Konrad stays lightweight and leans on well-maintained building blocks rather than custom code.** Prefer configuring an existing tool over shipping bespoke machinery; drive ongoing maintenance toward zero, so Konrad keeps working and current even through stretches when no one is actively developing it. (This is why model-discovery's custom plugin was dropped, the config deep-merge is a tiny self-contained step, and the whole pinning/rebuild story is a bot plus upstream registries.)

Its second face: **simple, logical, and smooth beats clever.** Konrad should stay tidy and resist sprawl. A solution that *feels* surprising (e.g. the runtime user lacking write access everywhere in its own home) is a smell to fix at the structural cause, not a quirk to live with — today's quiet workaround is tomorrow's unexpected consequence. Accumulating complexity is debt to pay down deliberately; periodic cleanliness passes are part of the maintenance story.

## Configuration & instructions

Config is **layered, not replaced** — host-mergeable layers, folded left, last writer wins:

```text
baked image defaults   <   org layers (0..n)             <   user
/etc/konrad/…               ~/.config/konrad/org/<name>/      ~/.config/konrad/user/
```

- Each layer holds the same opencode pieces — `opencode.jsonc`, `agents/`, `skills/`, `instructions/`, `AGENTS.md`, `fonts/`. [image/entrypoint.sh](image/entrypoint.sh) composes the `opencode.jsonc` of each via [image/merge-config.js](image/merge-config.js) (an N-input deep-merge) before opencode loads anything. **Objects merge recursively; arrays replace** — which is why agent rules are added via `AGENTS.md` or `instructions/`, not by overriding the `instructions` array. (A layer may also carry an `allowed_hosts` file — Konrad-specific, consumed by the egress firewall, *not* part of this opencode merge.)
- **`org/` is a container of named layers, and a subscribed layer is a managed git mirror.** `konrad org add <git-url>` clones a repo to `org/<name>/`; `konrad org sync` (riding every `konrad update`) does `git fetch` + `reset --hard @{u}` — local edits to tracked files are clobbered by design (the user's channel is `user/`, which always merges on top), a failed sync warns and keeps the last-good checkout (availability beats freshness for a defaults mechanism), and there is **no state file** — the in-place `.git` records URL and branch, so `org list` reads what sync will actually use. Layers fold in alphabetical name order (`10-`/`20-` prefixes steer precedence); a plain non-git dir is a valid manual layer that never syncs; a detached-HEAD checkout (a tag) is the pin mechanism. This is why an org ships updates by `git push`, with zero bespoke updater and **zero Konrad auth code** — private repos work through the member's existing git credentials (forge CLI login or SSH), auth by composition. The entrypoint, [compose-allowed-hosts.sh](image/konrad-defaults/compose-allowed-hosts.sh), and the overlay loops all walk the *same* sorted enumeration, so precedence can't drift between channels — and the firewall derivation sees **every** layer's providers (a missed layer would silently block that org's internal endpoint).
- **The `hooks/post-sync` hook is a deliberate exception to "agent code runs only in the sandbox."** A layer may ship an executable `hooks/post-sync`; Konrad runs it host-side after `add` and after every successful sync (cwd = the layer dir, no special env — it derives paths from the well-known `~/.config/konrad/` home, same principle as layer discovery). It exists for the ~10% of org needs plain config can't express (mirror a corpus into `context/`, derive per-member identity from the org's own forge auth) and it *rides the branch*, so it self-updates with the layer — a freestanding script Konrad merely pointed at would re-create the frozen-clone problem this design replaces. Trust model: **subscribing is trusting** — no prompt, no flag; the decision to add a layer is the decision to run its code, documented as the boundary. Context-mirroring stays hook territory only; Konrad gains no declarative corpus-sync engine.
- **Discovery is a well-known `$HOME` folder** (`~/.config/konrad/{org,user}`), not an env var or system path — because the macOS Podman machine only auto-shares `$HOME` into its VM, so a `/etc`-style path would be invisible there.
- **`AGENTS.md` is the user's slot; `instructions` is Konrad's.** opencode loads both, additively. The system `instructions` channel: the baked `opencode.jsonc` declares one glob for the baked `instructions/` dir and one for the user's, and opencode expands them itself — skipping absent dirs/files, de-duplicating, preserving order (per opencode's `Instruction.systemPaths`). **Org layers can't get a glob of their own**: opencode expands an entry by globbing only its *basename* component (the dirname is a literal), so `org/*/instructions/*.md` could never match — instead the entrypoint **copies** each layer's `instructions/*.md` (and back-compat layer-root `AGENTS.md`) into the baked `instructions/` dir as `org-<name>-<file>`, where the baked glob finds them. That's the same mechanism as the generated egress allow-list file, and uniform with the per-layer `cp -r` of agents/skills — no `jq`, no config surgery, and the array-replace footgun never bites because only the baked layer declares the `instructions` array. The user's global `AGENTS.md` stays theirs alone (opencode auto-discovers it). Precedence, all additive: `environment.md → org instructions → user instructions → user AGENTS.md → project AGENTS.md`.
- The org layer is **defaults, not enforcement**: files in the user's own home, so "add-only" describes merge precedence, not a permission lock (and sync clobbers local edits, it doesn't protect them).
- **No model auto-discovery, no baked `model`.** Provider endpoints ship pre-wired but model lists ship empty — users declare their loaded models in `opencode.jsonc`; opencode prompts for the model on first run and remembers it in the `konrad-state` volume.
- **Tool permissions are sandbox-shaped and live once, globally.** The baseline `permission` block sits in [opencode-defaults.jsonc](image/konrad-defaults/opencode-defaults.jsonc) (top-level → every agent inherits it via the same deep-merge); per-agent blocks carry **only deltas**, since they merge over it. The model follows the boundary, not the action's scariness: the container is disposable and non-root, so the agent acts freely *inside* it (`allow`), `ask` is reserved for the one irreversible action that hits the user's real files (`rm -rf` in the `/workspace` bind), and `deny` covers the escape vectors — root (`sudo`), host container/cluster control (`podman`/`docker`/`kubectl`), and the secrets volume under `external_directory`. Network tools stay `allow` because the [egress firewall](#egress-firewall) — not a per-command gate — is the real network boundary. Both `allow` and `deny` are prompt-free; only `ask` interrupts, so prompts are rare by design. Subagents add `question: deny` (no user in their loop) and tighten `rm -rf` to a hard `deny`; [manual-transformer](image/opencode/agents/manual-transformer.md) is the worked example.

## State, secrets & isolation

**`.agent/` belongs to the agent end-to-end; framework state lives elsewhere.**

```text
.agent/
  task.md                     # planning artifact (committable)
  artifacts/                  # durable mid-task outputs (committable)
  scratch/                    # agent scripts / probes
  quality-assurance/<stamp>/  # verification evidence on a fail
```

- **konrad doesn't touch the workspace at startup.** It neither creates the `.agent/` tree (the agent and the quality-assurance helper `mkdir -p` their own dirs on demand), nor prunes it, nor edits `.gitignore`, nor writes a session sidecar. The workspace bind is the agent's to shape — a launch leaves a clean repo clean.
- **Sessions are ephemeral.** opencode's `~/.local/share/opencode/` isn't bind-mounted — sessions, SQLite DB, and logs die with `--rm`. Durable task memory is `.agent/task.md`, not the session DB; collapsing the two parallel durability paths keeps the design coherent.
- **Secrets never touch the workspace** — `auth.json` lives only in the `konrad-secrets` named volume.
- **Logs land at a host XDG path** (`~/.local/state/konrad/log/`), bind-mounted narrowly so opencode keeps writing structured logs while the workspace stays pristine. No `konrad logs` subcommand — write to a documented path and trust `tail`/`grep`, like `npm`/`brew`.

| Mount | Purpose | Type |
|---|---|---|
| `konrad-secrets` ↔ `…/.opencode-secrets/` | `auth.json` | named volume |
| `konrad-cache` ↔ `…/.cache/opencode/` | regeneratable cache | named volume |
| `konrad-state` ↔ `…/.local/state/opencode/` | model choice, small UI state | named volume |
| `~/.local/state/konrad/log/` ↔ `…/opencode/log/` | opencode logs | host XDG bind |
| `~/.config/konrad/context/` ↔ `/context` | org/user reference material | host bind, read-only, when present |
| `<cwd>` ↔ `/workspace` | the user's project | workspace bind |

`--profile <name>` suffixes the state + cache volumes for throwaway self-test isolation; `konrad reset` wipes the volumes + log dir (not `context/` — that's user-managed material, like the config layers).

**The `context/` mount is a reference *substrate*, not a config layer.** A sibling of `org/`/`user/` under `~/.config/konrad/`, it's bound read-only at `/context` (named corpora in subdirs, `/context/<name>`) so the agent can `rg` org/user-supplied material — a host-mirrored wiki, internal docs — with no network and no stored secret. It carries none of the five opencode pieces and takes no part in the `baked < org < user` merge: it's *data the agent consults*, not *settings that shape the run*. So it's bound only when the dir exists, and — unlike the org/user config mounts — it's **agent-only**: it goes in `home_mounts`, never `cfg_mounts`, because the firewall proxy consumes `cfg_mounts` to derive the allow-list and has no business seeing reference material. A bind (not a baked-in skill) is the right primitive precisely because a corpus is large, churny (re-synced host-side, out of band), and zero-copy — a mount stays fresh for free, where baking it into a config-layer skill would shuffle it through `merge-config.js` on every launch. The choice of *bytes-as-bind* also enables the activation story to layer cleanly on top, matched to how hard the agent should reach for a corpus: passive (the `environment.md` `/context` hint), described (an org/user `AGENTS.md` naming the corpus), or active (a *thin* org skill that **points at** the mount — progressive-disclosure triggering without bundling the corpus). A skill that *contains* a corpus is the anti-pattern; one that *references* the mount is the premium path, and the mount is what makes it possible. (The named escalation, if a corpus ever outgrows `grep`: an MCP retrieval sidecar — deliberately out of scope here, since local grep over a bind is the lightweight, composed answer.) Mount *presence* is also the boundary the opt-in confidential tier (ROADMAP) builds on — "don't mount the dir" is a cooperation-free seal in a way "don't load a skill" is not.

**Resource caps are default-on and auto-scaled to the host, identical on both engines.** Every agent launch runs with a `--memory` / `--cpus` ceiling. When unset, the default is derived from the machine's own totals — half the host RAM (clamped to `2G`–`8G`) and all but two of the cores (clamped to `2`–`8`) — so a laptop and a workstation each get a ceiling that fits, never one exceeding what the host has (`host_mem_mib` / `default_memory` / `default_cpus` in [bin/konrad](bin/konrad), Linux `/proc` + macOS `sysctl`, best-effort with a fixed `4G`/`4` fallback). The `8G` RAM ceiling — and the matching 8-core CPU ceiling — are deliberately modest rather than an unbounded grab: docling fits comfortably in both, so a bigger machine keeps its headroom for a co-resident local model (reached via the proxy at `host.containers.internal`) instead of konrad claiming most of the box. The two ceilings are not symmetric in force, though: RAM is a hard allocation (over-commit means an OOM kill), whereas `--cpus` is a time-shared quota — an idle konrad withholds no cores, so the CPU ceiling only bounds konrad's share during a simultaneous compute burst (docling/torch, whose memory-bandwidth-bound thread scaling flattens before 8 threads anyway). Heavy OCR raises either cap explicitly. `KONRAD_MEMORY`/`KONRAD_CPUS` pin an explicit value; `0` disables a cap. The default does double duty: it lifts Apple `container`'s 1G per-VM default — too small for the bundled docling models, which the kernel OOM-kills mid-run — *and* bounds runaway/fork-bomb memory + CPU on otherwise-uncapped Podman. Both CLIs accept the same flag spelling, so this needs no `eng_*` wrapper. The CPU cap is also exported into the container as `OMP_NUM_THREADS` (floored to an integer; the host core count when the cap is off): docling/torch and any OpenMP/numpy work otherwise self-throttle to a 4-thread default no matter how many cores the cgroup allows, so granting cores without this leaves the image's heaviest CPU consumer idling most of them — the two halves ship together. (GPU acceleration is moot here: torch is the `+cpu` build and the engines run a Linux guest, so neither CUDA nor Apple MPS is reachable; `--device` stays `cpu`.) On the shared-kernel Podman path the agent and proxy are further hardened — every Linux capability dropped (`--cap-drop=ALL`), `no-new-privileges`, and a task-count ceiling (`--pids-limit`, default `1024`, `KONRAD_PIDS_LIMIT` overrides) that bounds a fork bomb below the host's PID max; apple/container already bounds these at its per-container VM boundary, so the block is Podman-gated. (Read-only-workspace hardening is still on the [ROADMAP.md](ROADMAP.md).)

### Egress firewall

**The agent has no direct route to the internet; a sidecar proxy is the only way out, and it forwards only an allow-list.** This closes the network as an exfiltration channel — a prompt-injected agent can't ship the workspace or the provider credentials to an arbitrary host.

- **Two networks, per run.** `bin/konrad` creates an `--internal` network (no route out) and a normal egress network. The agent joins *only* the internal one; the proxy joins both. So the boundary is enforced by the engine's networking, not by the agent's cooperation — the agent (uid 1000) can't reconfigure it. A Podman *pod* can't do this: pod members share one netns, so they'd share the same egress. Separate containers on separate networks is what isolates them.
- **Same topology on both engines (Podman and apple/container), but the apple/container plumbing is fussier** — all of it behind the `eng_*` wrappers, `fw_proxy_url()`, and `fw_setup()`. The deltas, each verified on macOS 26:
  - *Agent → proxy by IP, not hostname.* Podman serves the proxy's container name over its internal DNS even on an `--internal` net; apple/container has no resolver on a custom net, so the agent is handed the proxy's literal internal-net IP (re-read each run — it changes on recreate).
  - *Proxy egress leg is the built-in `default` net.* A `container network create` network on apple/container has **no DNS resolver and no NAT** — only the engine-managed `default` net resolves and reaches the internet. So the proxy is dual-homed on the custom internal net (to face the agent) **plus `default`** (to get out); Podman keeps a dedicated, torn-down egress net since every netavark network resolves+NATs.
  - *Attach order is load-bearing.* A container inherits its resolver from the **first** `--network`; an `--internal` net contributes none, so `default` must be listed **first** or DNS dies even though it's attached. The agent stays internal-only on purpose — it never resolves anything (it only ever talks to the proxy by IP).
  - The security property is identical to Podman's: enforcement lives in a container the agent can't reach, surviving a root-level agent compromise.
  - **Host-gateway seal (apple/container only).** apple/container's `--internal` net is *host-only*: its vmnet gateway sits on the Mac, so a host service bound to `0.0.0.0` would be reachable **directly** from the agent at the gateway IP, bypassing the proxy and the allow-list (rootless Podman has no such route — a container on a podman `--internal` net gets *network unreachable* to the host, verified macOS + Linux). Konrad seals it: on the apple/container firewall path `bin/konrad` starts the agent as root with `CAP_NET_ADMIN` and hands its entrypoint the internal-net gateway, which the root prelude `ip route add blackhole`s before dropping to the unprivileged `node` user with the capability stripped (shared `exec_as_node` in [image/konrad-privdrop.sh](image/konrad-privdrop.sh)). The blackhole is kernel state the dropped agent can't remove, so the agent ends up both sealed and capability-less — and apple/container reaches **parity with Podman**: the only remaining path to the host is back through the proxy, at the allow-listed `host.containers.internal`. The seal is **fail-closed** — if the gateway can't be read or the route won't install, the run aborts rather than expose the leak (`--no-firewall` or `KONRAD_ENGINE=podman` knowingly opt out).
  - **Local-model host alias (apple/container only).** Sealing the direct path means local models route through the proxy like any other allow-listed host — exactly as on Podman. The one missing piece is name resolution: Podman injects `host.containers.internal` into the proxy's `/etc/hosts` for free; apple/container injects no such alias and has no `--add-host`, so `bin/konrad` hands the proxy the egress (`default`) net's gateway — the Mac host — and the proxy's root prelude (the same `konrad-entrypoint` block, keyed on `KONRAD_HOST_ALIAS_IP`) writes that mapping into `/etc/hosts` before dropping to `node`. `host.containers.internal` is in the baked allow-list floor, so a model request the agent forwards through the proxy reaches the host; any *other* host name or raw gateway IP stays default-denied.
- **A forward proxy, on purpose.** The agent gets `HTTP(S)_PROXY` pointed at the sidecar. opencode runs on Bun, which honours those env vars for `fetch`; this was verified end-to-end (its HTTPS model-catalog `CONNECT` and local-model HTTP calls both route through). A forward proxy keeps the agent capability-less — no `NET_ADMIN`, no nftables, no transparent-interception machinery. The proxy is [tinyproxy](https://tinyproxy.github.io/) (`apt`-installed into the one image, not a second artifact to pin/pull) with `FilterDefaultDeny` + an anchored host filter.
- **The allow-list is derived, not hand-maintained.** The proxy runs the *same* `merge-config.js` over the *same* baked < org < user layers the agent sees and extracts every `provider.*.options.baseURL` host. opencode's *built-in* providers carry no `baseURL` (you enable them with just a key or `/connect`; opencode resolves their endpoint from the SDK / models.dev), so for those the proxy resolves each provider the user has **connected** (`auth.json` — read-only, ids only, never the keys) or **declared** (a `provider` block / a `model` prefix) to a host through a baked `provider-id → host` map ([image/konrad-defaults/provider-hosts.json](image/konrad-defaults/provider-hosts.json), generated from models.dev by [scripts/resolve-provider-hosts.sh](scripts/resolve-provider-hosts.sh) and refreshed by the daily lock bot — the same "pin upstream data, bake the pinned copy" pattern as `image/locks/`). So it tracks the user's real providers automatically, whether declared in config or picked interactively. Unioned with a deliberately tiny baked floor — `host.containers.internal` (local models) and `registry.npmjs.org` (the on-demand provider SDK adapters opencode isn't already bundling) — plus the org/user `allowed_hosts` files (+ `--allow-host` for a run). The floor was trimmed empirically: a configured model resolves and runs without `models.dev`, and OpenAI-compatible providers need no npm fetch, so `models.dev`/PyPI/the open web are opt-in, not default. Running the proxy *from Konrad's own image* is what makes the derivation free — a stock proxy image would have neither the merge tool nor the config layers. Host-based filtering (not IP) is deliberate: remote providers sit behind rotating cloud IPs. The derivation itself lives in **one baked script** ([image/konrad-defaults/compose-allowed-hosts.sh](image/konrad-defaults/compose-allowed-hosts.sh)) so it can't drift between consumers: the proxy anchors its output into the tinyproxy filter, and [image/entrypoint.sh](image/entrypoint.sh) inlines the *same* list into the agent's `instructions` at session start — so the model knows which hosts a fetch can reach instead of burning turns on ones the proxy will refuse. The proxy stays the enforcing boundary; the inlined copy is a start-of-session hint (it doesn't follow a mid-session `/connect` — the proxy's live-reload does — and it's only emitted when the firewall is on).
- **Live-reload, so connecting a provider needs no restart.** The filter isn't frozen at proxy start: the sidecar polls `auth.json` and, on change, rewrites the filter and signals tinyproxy (`SIGUSR1` — reloads the filter list without dropping active connections). So an in-session `/connect` of an API-key provider takes effect within seconds, no relaunch and no `--no-firewall`. The one exception is an *OAuth* first-connect (Claude Pro/Max, Copilot): that handshake needs the host *before* `auth.json` exists, so a mid-session connect can't auto-allow it. The clean path is **`konrad connect`** (`do_connect` in [bin/konrad](bin/konrad)) — it runs `opencode auth login` with the firewall **off**, which is safe precisely because no agent is in the loop (pure authentication, no prompt-injection surface), so OAuth and API-key logins alike need no flags. A one-time `--allow-host`/`--no-firewall` remains the escape hatch for connecting *inside* a normal session.
- **Default-on, with a clean bypass.** On for every `run`/`shell`/`run`-oneshot launch; `--no-firewall` (or `KONRAD_FIREWALL=0`) restores the pre-firewall unrestricted path (a bare `exec podman run`). The proxy + networks are per-run (`$$`-named) and torn down by an EXIT trap. Mechanism lives in [image/konrad-proxy-entrypoint.sh](image/konrad-proxy-entrypoint.sh) and `fw_setup`/`fw_teardown` in [bin/konrad](bin/konrad).

## The planning contract

Two **independent** gates, baked into the agent prompt ([image/opencode/agents/konrad.md](image/opencode/agents/konrad.md)):

- **`.agent/task.md`** — when the task has *side effects* (file edits, state-changing commands). A binary check at first tool-selection, not a complexity prediction. The agent writes it *before* the first side-effecting call, in a fixed shape:

```markdown
# <task title>
## Understanding       — what the user wants, as I read it
## Plan                — 3–5 bullets (the path, not micro-steps)
## Success looks like  — how I'll know I'm done
## Decisions & findings — appended during execution
## Outcome             — what shipped, what didn't, caveats
```

- **`todowrite`** — for anything beyond a trivial single Q&A; opencode's live in-UI checklist.

`task.md` holds *what & why* (durable, survives context compaction); `todowrite` holds *where the agent is right now*. No overlap, no sync burden. The agent surfaces the plan inline and proceeds when confident, or asks via `question` when a wrong guess would be costly (ambiguous goal, irreversible step). The `quality-assurance` skill later reads `task.md`'s Plan + Success + Outcome to render its verdict.

## Build & reproducibility

**Every input is locked; a bot maintains the locks; the publish is smoke-gated.**

- Each input is digest- or version-locked in [image/locks/](image/locks/): base image, the `uv` source image, Python deps (`python.lock`), `opencode-ai`, `typst`, and the Docling model repos (`models.lock` — commit-pinned so the ~1.1 GB model layer stays byte-stable across rebuilds; one `COPY` layer per model means a single-model update re-pulls only that model, not the whole set). apt packages float — they refresh whenever the base image bumps. The [image/Dockerfile](image/Dockerfile) top comment mirrors this list.
- The Python venv ships as **three `COPY` layers** (the `venv-split` build stage partitions it): torch (~700 MB), the large slow-moving numeric wheels (opencv/scipy/numpy/onnxruntime/sympy, ~450 MB), and the churny remainder (~360 MB). So a `python.lock` bump that only moves a small pure-python dep — the near-daily case, since the bot tracks every transitive — re-pulls just the light layer, while torch and the numeric set dedupe (same byte-stability mechanism as the per-model split above). Sibling to the model-layer dedupe.
- The GitLab `resolve-locks` bot re-resolves each upstream daily and opens an auto-merging MR **only when something moved** — so a rebuild fires only when there's genuinely something new. No scheduled cron rebuild. (The build-side of the low-maintenance ethos: freshness is a bot plus upstream registries, not hand-tended bumps.)
- **Reproducible layers** — CI builds with `SOURCE_DATE_EPOCH=0` + `rewrite-timestamp=true`, so byte-identical content yields byte-identical layer digests; that's what lets users skip a re-download when nothing actually changed. Local `konrad-dev rebuild` writes `konrad:local`, separate from the published `:latest`.
- **CI runs on a GitHub mirror; the primary repo stays on `gitlab.git.nrw`** (one-way push mirror → GitHub Actions does build → smoke → publish to `ghcr.io/jlbauss/konrad`). Why: gitlab.git.nrw's shared runners can't run the privileged Podman build. The GitHub side is a CI execution surface only — no issues, no MRs. (Full contributor + release flow: [CONTRIBUTING.md](CONTRIBUTING.md).)
- Every image carries `/etc/konrad/build-manifest.json` (dpkg / npm / pip versions + build metadata); `diff` two `:<short-sha>` images' manifests to bisect a regression to its upstream cause.

**AGPL-3.0** — compatible with all bundled upstream licenses (MIT, Apache-2.0, OFL-1.1). Strong copyleft is deliberate for a sandbox-style tool: AGPL's network-use clause closes the SaaS loophole plain GPL leaves open, so a fork offered as a hosted remote agent over an API must still publish its source. Per-file copyright + license is declared machine-readably following the [REUSE](https://reuse.software) spec (`REUSE.toml` + `LICENSES/`, `reuse lint`-verified); acknowledgements (the runtime, the model, adapted prompt-patterns) live in the [README](README.md#license-and-attribution).
