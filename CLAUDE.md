# CLAUDE.md

## What this repo is

konrad is a CLI (`bin/konrad`) that runs [opencode](https://github.com/sst/opencode) inside a sandboxed Podman container. The container image is the canonical artifact; the CLI is the primary consumer. The experimental `devcontainer/` is a second, not-yet-ready consumption path (see ROADMAP.md).

## Validation (no test suite)

There are no automated tests. Validate changes with:

```sh
shellcheck bin/konrad image/entrypoint.sh scripts/*.sh   # lint — should stay clean
bash -n <script>                                          # parse check
./scripts/build-image.sh                                   # smoke test: image actually builds
cd /tmp/konrad-test && konrad --version && konrad --shell  # live smoke test
```

## Development loop

1. Edit files
2. `./scripts/build-image.sh` (or `konrad-dev --rebuild` if the konrad-dev symlink is set up)
3. Run `konrad-dev` or `konrad-dev --shell` to verify against the dev image

Any change to `image/` (Dockerfile, entrypoint, defaults, agents) requires a rebuild before it takes effect. Changes to `bin/konrad` take effect immediately (it's a bash script).

## Repo layout and config (the canonical sources)

Repo layout is canonical in [README.md](README.md). The single rule worth carrying in your head while editing: **anything under `image/` requires a `konrad-dev --rebuild` before changes take effect**; `bin/konrad`, `scripts/`, and `.devcontainer/` are live (no rebuild). The "Skill / agent / instructions edits only take effect after `konrad-dev --rebuild`" working agreement below restates this for the cases agents hit most.

Config layering is canonical in [README.md](README.md). As of 0.4.0 there are three host-mergeable layers — **baked < org < user** (`/etc/konrad/opencode-defaults.jsonc` < `~/.config/konrad/org/` < `~/.config/konrad/user/`), folded left by [image/merge-config.js](image/merge-config.js). The constraints worth remembering when editing baked config under `image/konrad-defaults/`: **objects merge recursively (later layer wins), arrays replace entirely**. The array-replace rule is why users (and orgs) add agent rules via `AGENTS.md` instead of the `instructions` key — an override of `instructions` would silently discard the baked content. The one exception is **org `AGENTS.md`, which the entrypoint appends to `.instructions` *after* the merge** (via `jq`), precisely so array-replace can't drop it — that's the org layer's "system channel." Design rationale: [docs/design/org-config-layer.md](docs/design/org-config-layer.md).

Multi-concern changes: prefer separate commits per concern (see [CONTRIBUTING.md](CONTRIBUTING.md)).

## Important constraints

- **Podman only** — `--userns=keep-id` is Podman-specific. Docker support is on the roadmap.
- **No model auto-discovery.** Provider endpoints ship pre-wired but model lists ship empty — users declare their loaded models in `~/.config/konrad/opencode.jsonc`. The `opencode-models-discovery` plugin used to do this automatically; removed because of startup cost and an upstream embedding-modality bug. An inline replacement is on the roadmap.
- **No top-level `model` key in baked defaults** — opencode prompts on first run and persists the choice in the `konrad-state` named volume.
- **`.agent/` belongs to the agent end-to-end.** Layout: `.agent/task.md` (planning artifact, committable) · `.agent/artifacts/` (durable mid-task outputs like `manual-output.<ext>`, committable) · `.agent/scratch/` (agent-written scripts/probes, auto-pruned >7d, gitignored) · `.agent/quality-assurance/<stamp>/` (verification evidence on a fail, auto-pruned >7d, gitignored). No more `.agent/opencode/` — opencode session state is now ephemeral inside the container; logs land in `~/.local/state/konrad/log/` on the host. See [docs/design/state-isolation.md](docs/design/state-isolation.md) for the contract.
- **Python venv** at `/opt/venv` (on PATH). Extend with `uv pip install <pkg>`.
- **Debian renames**: `fd` → `fdfind`, `bat` → `batcat` (symlinked to canonical names in Dockerfile).
- **opencode Zen disabled** by default (`disabled_providers: ["opencode"]`).
- **Bundled skills** live at `image/opencode/skills/` (currently: `do-it-manually`, `spreadsheets`, `pdf`, `quality-assurance`). They land at `~/.config/opencode/skills/` in the image and are loaded by opencode's `skill` tool. `quality-assurance` is the cross-skill verification cycle (visual + language) every producer skill invokes before reporting.
- **Bundled fonts.** Seven OFL families ship at `image/fonts/konrad/` → `/usr/local/share/fonts/konrad/` (Inter, Source Serif 4, Fraunces, JetBrains Mono, EB Garamond, IBM Plex Sans, Atkinson Hyperlegible). Fontconfig-aware tools (Typst, Playwright, LibreOffice) discover them automatically; reportlab needs `font_helpers.register_font(family)` from the pdf skill. Users add their own by dropping files into `~/.config/konrad/fonts/`. Catalogue in [image/opencode/skills/pdf/references/fonts.md](image/opencode/skills/pdf/references/fonts.md).

## Commit style

Conventional commits without scope prefix unless useful: `feat:`, `fix:`, `refactor:`, `docs:`, `chore:`. Multi-line body for any change involving a design decision — the git log is the project's design history.

## Working agreements

Durable conventions for any agent working in this repo. Written here (rather than only in per-session agent memory) so every Claude instance sees the same rules.

- **When a ROADMAP item ships, record it in `CHANGELOG.md` and delete the tier bullet — same commit.** Add a terse, user-facing entry under the right version heading in [CHANGELOG.md](CHANGELOG.md) ([Keep a Changelog](https://keepachangelog.com/) — `### Added` / `Changed` / `Fixed`), and remove the bullet from its tier (`## Tier 0/1/2`). The changelog is **hand-maintained by the agent** (no generator): keep entries terse and let the commit body + [design docs](docs/design/) carry the *why* — never restate rationale in the changelog. Unreleased work goes under `## [Unreleased]`; it rolls into a version heading when `VERSION` bumps. Doc-only / CI-only / contributor-only changes that don't bump `VERSION` usually don't warrant a changelog entry (they live in git). If a tier item only partially ships, changelog what landed and spin the remainder into a fresh tier bullet so nothing falls off the radar.
- **Roadmap edits use the `docs(roadmap)` prefix.** Editing [ROADMAP.md](ROADMAP.md) to add/reword backlog ideas, shuffle bullets, or promote inbox items is a sync action, not a narrative one — the rationale lives in the bullet itself (and, for shipped work, in the commit body + [CHANGELOG.md](CHANGELOG.md)), not in the commit message. **When the user commits**, a bare `docs(roadmap)` subject is enough (that's what they usually type), or they edit ROADMAP.md in the GitLab web UI (auto-fills the message, commits to `main`). **When an agent commits**, append a small note on *what changed* so the log is scannable without opening the diff — e.g. `docs(roadmap): promote org-config layer to Tier 1; drop shipped badge item` — one line, no body; the bullet still carries the substance. Batch a session's roadmap edits into one commit and fold follow-up tweaks in with `git commit --amend` **before** pushing — never amend after push, since the user syncs across multiple devices and already has it. Roadmap-only changes trigger no image build (the `image/**` path filter), so these commits are cheap, and the `docs(roadmap)` prefix keeps them filterable out of a feature-history view. This does **not** apply to `CHANGELOG.md` entries that ship alongside real work — those ride the feature commit and keep its descriptive message.
- **Doc-surface contract: one audience per file.** Each repo doc has one primary audience: [README.md](README.md) → end users; this file (CLAUDE.md) → agents working on konrad's source; [CONTRIBUTING.md](CONTRIBUTING.md) → human contributors; [ROADMAP.md](ROADMAP.md) → backlog tiers; [CHANGELOG.md](CHANGELOG.md) → the released-change log (terse, Keep a Changelog); [docs/design/](docs/design/) → long-form rationale for individual design decisions (the *why* behind non-obvious choices). Runtime content is separate from these repo docs and serves opencode at runtime, not human readers: [image/opencode/environment.md](image/opencode/environment.md) is the konrad agent's environment manifest (tool inventory, Python libraries, filesystem layout, locale, constraints), [image/opencode/agents/*.md](image/opencode/agents/) holds the agent behavioral prompts, [image/opencode/skills/*/SKILL.md](image/opencode/skills/) holds per-skill workflow descriptions. User overlays at `~/.config/konrad/user/AGENTS.md` and `<workspace>/AGENTS.md` are **additive** to the baked instructions, never replace them; an org's `~/.config/konrad/org/AGENTS.md` is likewise additive but rides the `instructions` channel (see the config-layering note above). **The rule**: when in doubt about where something belongs, name the audience first; if a fact serves two audiences, write it once in the canonical place and link from the other — never duplicate.
- **Important wishes from the user belong in this file.** When the user gives a durable rule, preference, or convention to apply across sessions, write it here as a Working agreements bullet (in addition to any agent-internal memory). This file travels with the repo, so every agent on every machine picks it up — agent-side memory does not. This is the canonical-instance application of the Doc-surface contract above (durable user preferences serve agents-on-source, so they live in CLAUDE.md).
- **Prefer `trash` over `rm` for deletes inside `/workspace`.** `trash <file>` (from trash-cli, installed in the dev container) moves the file to `/workspace/.Trash-1000/files/` on the host filesystem — survives container rebuilds and is recoverable with `trash-restore`. `trash-list` shows what's there; `trash-empty` purges. `rm` still hard-deletes; use it deliberately. VS Code's right-click "Delete" in the Explorer also hard-deletes for `/workspace` files (a known limitation of VS Code's remote-trash code path in dev containers as of 2026-05-18 — the underlying mechanisms work but VS Code's wrapper around them errors out for /workspace). Use the terminal `trash` command for the safety net.
- **Say "quality assurance," not "QA."** Durable terminology preference: in prose anywhere under `image/opencode/` (skill docs, agent prompts, instructions) write the full phrase. File and directory names follow the same convention where the format allows it: `quality-assurance.md` (kebab-case to match `do-it-manually` etc.); Python module names use snake_case (`quality_assurance_helpers.py`); the user-visible state path is `/workspace/.agent/quality-assurance/<stamp>/`. The full phrase reads as self-explaining where "QA" is jargon — that's the load-bearing reason; commit messages, reports, and changelog entries all follow it.
- **Skill / agent / environment edits only take effect after `konrad-dev --rebuild`.** Anything under `image/opencode/` (skills, agents, environment.md) and `image/konrad-defaults/` is baked into the container image at build time, not bind-mounted from the host. Editing a skill's `SKILL.md` or a `scripts/*.py` on the host changes nothing for the running konrad agent until the image is rebuilt. When iterating with the user on a skill, remind them to `konrad-dev --rebuild` between change and test — easy to forget and leads to confusing "but I changed it" moments. The same holds for `image/Dockerfile` (a rebuild is what installs new Python deps into `/opt/venv`). Standard dev loop: `konrad-dev --rebuild && konrad-dev` (konrad-dev defaults to `konrad:local`, so no env var dance). The stable `konrad` on PATH is unaffected.
- **Keep `bin/konrad` (and the other tracked scripts) executable after editing.** `bin/konrad` is the dev binary — `konrad-dev` symlinks to it, and bash/fish refuse to run it without the `+x` bit (`exists but is not an executable file`). Git tracks it as mode `100755`, so a clean checkout is fine; the bit gets lost when an *in-place rewrite* lands a fresh inode at the default `644` umask — an editor/agent save via temp-file-then-rename, or a host-side format-on-save. That silently breaks `konrad-dev` until a manual `chmod +x`, a recurring, confusing papercut in the dev loop (hit 2026-06-08). **So any agent that edits `bin/konrad` must `chmod +x bin/konrad` before finishing** (verify with `ls -l`). Same for the other tracked executables — `image/entrypoint.sh`, `scripts/*.sh` — if a rewrite touches them. If it keeps recurring from the *user's own* editor, the host-side fix is to disable format-on-save for shell scripts or add a `post-checkout`/`post-merge` hook that re-chmods.
- **Bump `VERSION` in the same PR as the change.** The `VERSION` file at the repo root drives all image tagging (`:<ver>-<date>`, `:<ver>`, `:latest`). Pre-1.0 it holds `0.X.Y`. Bump **`Y` (patch)** for a bug fix with no surface change — same flags, same outputs, same behavior on the happy path. Bump **`X` (minor)** for new functionality, behavior change, base-image major bump, or anything a user might notice; reset `Y` to 0. Doc-only, CI-only, and ROADMAP-only changes don't bump. Reviewer checks the bump matches the change's semver impact. **Time the bump as the last commit before you merge, not up front** — with ff-only merges the branch tip *becomes* `main`, so a late bump keeps the version reflecting the integrated change and shrinks the window where two in-flight branches collide on the `VERSION` line, all without giving up the author-makes-the-semver-call benefit. Post-1.0 follows full semver — MAJOR for breaking, MINOR for additive, PATCH for fixes; full design in [docs/design/versioning-and-releases.md](docs/design/versioning-and-releases.md). **The README version badge auto-derives from `VERSION`** — it's a shields.io *dynamic* badge that reads `VERSION` over the public GitLab raw URL (the same one [bin/konrad](bin/konrad) uses for its update check), so it needs **no** manual bump and must **not** be replaced with a hardcoded `version-0.X` badge. If that badge ever has to revert to static (shields outage, repo goes private and the raw URL stops resolving anonymously), the fallback rule is: bump the badge string in the same commit as `VERSION`.
- **CLI and image versions can drift on CLI-only patches; that's fine.** `VERSION` drives both the CLI's baked value and the image's `org.opencontainers.image.version` label, but the CI image rebuild only fires when `image/**` / `scripts/smoke-test.sh` / the workflow itself moves — *not* on `VERSION` alone. So a CLI-only patch (bash-only fix, `install-remote.sh` tweak) bumps `VERSION`, ships via the next pull of the CLI, and leaves the image's label at whatever `VERSION` was at the last image rebuild — until the next image-relevant change ratchets it forward. `konrad --version` prints both numbers (`CLI:` and `image:`) for transparency; **no skew warning, no nag** — patch-line drift is normal and unambiguous. The "Konrad is one artifact" line is a *release-time* convention (bump together in the same PR), not a runtime invariant — the CLI/image contract is thin (a handful of env vars, mount paths, the `opencode` entrypoint) and tolerates a patch of drift. Why: rebuilding 412 MiB / 23 layers for a one-character bash fix wastes CI minutes and registry bandwidth without changing anything inside the image. If a CLI change ever genuinely depends on a new image (rare — would require a new env var the entrypoint must read, or a renamed mount), bundle both in one commit so `image/**` triggers the rebuild and the labels re-converge.
- **Trunk-based development on GitLab; merge via git CLI, not via MR/PR.** Primary repo is `gitlab.git.nrw/jbauss2/konrad`; GitHub (`github.com/jlbauss/konrad`) is a one-way push mirror that exists only as a CI execution surface. Everyday flow: branch off `main` on GitLab (`feat/<short>`, `fix/<short>`, `docs/<short>`, `chore/<short>`), commit, push the branch, then merge locally — `git checkout main && git merge --ff-only feat/<short> && git push origin main && git push origin --delete feat/<short>`. GitLab mirror copies main to GitHub within ~1 min; GitHub Actions sees the push and runs `build-image.yml` (build → smoke → publish under `:0.X.Y-<date>` / `:0.X.Y` / `:0.X` / `:latest` / `:<sha>`). Smoke is publish-gated, so a broken merge stops at `:latest` (which stays on the previous good build) while main still carries the broken commit — recoverable via revert. The design rationale lives in commit messages, not MR/PR descriptions; this is a solo workflow and the MR overhead is noise. **For higher-risk changes**, optionally push the feature branch to GitHub too and open a GitHub PR there for pre-merge `ghcr.io/jlbauss/konrad:pr-<num>` testing (`KONRAD_IMAGE=ghcr.io/jlbauss/konrad:pr-<num> konrad`); merge on GitLab once the PR image works. GitLab MRs are also available if you want a review thread on something specific. See [CONTRIBUTING.md](CONTRIBUTING.md) for the full workflow.
- **Validate smoke locally before pushing CI changes.** Editing `scripts/smoke-test.sh`, `image/Dockerfile`, or `image/build-manifest.sh`? Run `konrad-dev --rebuild && ./scripts/smoke-test.sh konrad:local` before pushing. CI round-trips cost more than the local loop. Cross-reference Debian package names vs binary names when listing things to check (`ripgrep`→`rg`, `fd-find`→`fdfind`, `bat`→`batcat`).
- **Self-testing the konrad runtime from the dev container.** The `.devcontainer` mounts the host's rootless podman socket (`CONTAINER_HOST`), so `podman` in here drives the host daemon — you can exercise the runtime image without round-tripping through the user. **Works cleanly:** `konrad-dev --rebuild` (build context streams to the daemon), `konrad-dev --shell` to inspect the image (`python3 -c "import docling"`, fonts, `/opt/venv`), and `konrad-dev` runs that touch workspace files ([bin/konrad](bin/konrad)'s `remote_host_path()` rewrites the `/workspace` mount source to its real host path in remote mode). `./scripts/smoke-test.sh konrad:local` works except its org-layer section (it bind-mounts host `mktemp` dirs the host daemon can't see). **Limits in remote mode:** host-home mounts are skipped, so no host-side log capture and no config-layer overlay while self-testing; the dev-container konrad shares the host konrad's named volumes, so a live model configured on the host (drop API keys via the host's konrad) works in here too. **Prereq (Linux):** `systemctl --user enable --now podman.socket` on the host, or the socket mount fails and the container won't start. **Guard-rail:** that socket is full host podman control — don't `podman run --privileged` or mount host paths from in here; `podman push` stays gated (`ask`). Treat the host daemon with the care you'd give the host shell.
- **When you actually invoke the konrad agent end-to-end (a live `opencode run`, not just `--rebuild`/smoke), use Qwen3.6 27B via OpenRouter — and pass it explicitly per run, don't inherit the host's persisted model.** Self-test runs and the (future) skill-A/B harness should pin one cheap, fast, consistent model: it avoids burning the user's default (often premium) selection, and it keeps A/B comparisons from being confounded by model drift. Override the model on the one-shot run (`opencode run --model openrouter/qwen/qwen3.6-27b "<prompt>"`) rather than relying on the model stored in the shared `konrad-state` volume — that selection is the user's, and mutating shared self-test state is to be avoided anyway (an end-to-end run against the real `konrad-state` volume triggers opencode's one-time DB migration on it; prefer isolated `konrad-selftest-*` volumes once a non-interactive `konrad` run mode exists — see ROADMAP). OpenRouter must be configured with a key in the secrets volume for the slug to resolve.
- **Prefer dedicated tools over shell for inspection; keep Bash calls statically allow-able.** The committed [.claude/settings.json](.claude/settings.json) allow-list lets benign commands run without prompting — but the permission matcher only auto-allows a Bash call it can *statically* prove safe, and two things defeat that even when the base command is allow-listed: **command substitution** (`$(…)` / backticks — the matcher can't know what it expands to) and **long compound chains** (`a ; b || c` — evaluated as a unit, one unprovable segment prompts the whole thing). So `grep` is allowed but `ls "$(id -u)/foo"` prompts — the `$(…)` is the trigger, not grep. **Two-part discipline to keep the flow without weakening security:** (1) use the **Read / Grep / Glob** tools for inspection instead of `cat`/`grep`/`ls`/`find` in Bash — they're first-class allowed and never prompt; reserve Bash for what only a shell can do (running konrad, git, builds). (2) When Bash *is* required, write one allow-listed command per call — avoid `$(…)` and `||` fallback chains; if you need a dynamic value, let it come from the environment or split into steps rather than computing it inline. **Do not widen the allow-list to cover `$(…)`** — that's the same injection surface the secret-read `deny` rules guard; the fix is agent discipline, not a looser list. When a *specific* read-only command prompts repeatedly and is genuinely safe, add it as a precise `Bash(<cmd>:*)` allow rule (the `fewer-permission-prompts` skill automates spotting these) — narrow additions are fine; blanket ones are not.
- **"Konrad" the name, `konrad` the code.** The project / agent / product is **Konrad** with a capital K — it's a name; write it capitalized in prose (README hooks, design docs, agent-facing text). The command users type, the published image, the package, and every filesystem path stay lowercase `konrad` — that's code and must not be recapitalized (`bin/konrad`, `ghcr.io/jlbauss/konrad`, `~/.config/konrad/`, `konrad --update`). Rule of thumb: thing-as-a-product → "Konrad"; literal command/path/image → `konrad` in backticks.
- **Keep Konrad lightweight; compose, don't custom-build; aim for near-zero maintenance.** The maintainer's standing engineering ethos: prefer stacking well-maintained existing building blocks (upstream tools, registries, BuildKit, opencode itself) over writing and owning bespoke machinery. When a capability can be had by configuring an existing component instead of shipping code, take the configuration. The goal is to drive ongoing maintenance burden toward zero — Konrad should keep working and stay current even through stretches when nobody is actively developing it (this is *why* the freshness story is a lock-resolver bot plus upstream registries, not hand-tended bumps). Follow established best practices rather than inventing local ones. Weigh any proposal that adds custom code or a new maintenance surface against this bar, and flag it when a change works against it. Recorded in user-facing form at [docs/design/design-decisions.md](docs/design/design-decisions.md#engineering-ethos--lightweight-composed-low-maintenance).
- **Keep it simple, logical, and smooth — resist sprawl.** The second face of the lightweight ethos above, and a standing wish from the maintainer: Konrad should stay *tidy*. Prefer the obvious shape over the clever one. Treat a solution that *feels* illogical or surprising (the canonical example: the runtime `node` user not having write access everywhere in its own home) as a smell to fix at the source, not a quirk to work around — because today's quiet workaround is tomorrow's unexpected-consequence, which is exactly what the maintainer wants to avoid. When an accommodation starts to feel like whack-a-mole (naming each intermediate dir to dodge a permission default, say), stop and name the structural cause; flag it even if the immediate patch ships. Accumulating complexity is debt to pay down deliberately — periodic cleanliness passes are part of the maintenance story (there's a Tier-2 roadmap item for one). When a change adds surface or a non-obvious mechanism, weigh it against this bar and say so. User-facing form: [docs/design/design-decisions.md](docs/design/design-decisions.md#engineering-ethos--lightweight-composed-low-maintenance).
- **Don't commit until the user has read your explanation.** When a change warrants a write-up (a diagnosis, a design rationale, a multi-file edit), present the full explanation first and let the user read it before calling the commit tool — don't chain the `git commit` into the same turn as the reveal. The user reviews the reasoning and the diff before it lands in history, and may want to adjust the approach or the message. Make the edits, explain them, then **wait for the user to say go** before committing. (This does not apply to trivial, already-agreed mechanical commits where the user has explicitly said to just commit.)
