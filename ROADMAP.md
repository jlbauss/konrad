## Tier 1 — alpha → beta

_Work done after the repo goes public as alpha. Gates the beta declaration._

- [ ] **Image-pull dedupe (b) — decouple the Docling model layer from the Python venv via a model lock.** Part (a) landed (see Implemented 2026-05-24); this is the follow-up. `/opt/models` is ~2 GB baked at [Dockerfile:180](image/Dockerfile#L180). Today stage 2 inherits from `python-base`, so any Python dep bump rehashes the model layer even when the model files themselves are byte-identical — a fake dependency, since the weights don't actually depend on torch's version. Fix: commit a `models.lock` listing each weight's source URL + expected SHA. Stage 2 becomes a thin stage that reads the lock and fetches weights directly, with no dependency on the venv — cache key is the lock contents. When `docling-slim` bumps but its required model set didn't change (the common case), the lock is unchanged and the model layer hits cache. When the model set genuinely changes, a small CI job diffs `docling-tools models download --dry-run` against the committed lock and opens a PR to update it; the layer then rebuilds legitimately. A BuildKit cache mount on the HuggingFace download path saves CI bandwidth on those rebuild days. Keeps konrad's offline-after-pull property intact — chosen over moving models to a runtime volume (which would shrink the image but break first-launch-offline and add staleness-check complexity to the entrypoint).

## Tier 2 — beta → 1.0

_Work done after the repo goes public as beta. Mostly speculative or aspirational._

- [ ] **Docker support.** Currently Podman-only because of `--userns=keep-id`. A Docker-compatible alternative `devcontainer.json` (or conditional `runArgs`) would broaden the audience.

They deliberately don't cover whether the output *looks good*. A separate design skill owns visual quality — typography, layout, color, hierarchy, brand consistency — and composes with the technical skills when the user wants polished output rather than just functional output. Open scope question: advisor that critiques drafts, generator that produces themed templates, or both. Pairs with the "no AI slop" aspiration: a deck that's structurally correct but aesthetically broken is still slop.
- [ ] **More skills.** `.eml` email, Markdown authoring, HTML presentations, odcast + audiobook creation skill
- [ ] **Inline model discovery (replaces the dropped plugin).** Eventually replace the removed `opencode-models-discovery` plugin with ~30 lines of host-side bash in `bin/konrad` plus a small `jq` step in `image/entrypoint.sh` that writes the discovered providers as a runtime config override. Empirically the plugin cost ~3.7 s of Bun startup; doing it natively eliminates that without giving up the feature. Only worth doing once we've also resolved the LM Studio embedding-modality issue upstream (or accept the same exclusion in our inline version).
- [ ] **Container hardening (resource limits, network isolation, read-only mounts).** Three complementary flags for `podman_run()` in `bin/konrad`: (1) `--memory` / `--cpus` / `--pids-limit` — cap RAM, CPU shares, and process count to prevent runaway or fork-bomb scenarios; (2) `--network=none` — remove the network stack entirely so the container cannot reach the host's local ports or any external address; combine with a narrow allow-list (e.g. `--network slirp4netns:...` or a sidecar firewall) only for the specific provider endpoints that need to be reachable. (3) Mount the workspace read-only by default (`-v $WORKSPACE:/workspace:ro`) and provide a dedicated writable output volume for agent writes — prevents an agent from accidentally (or intentionally) modifying source files it was only asked to read. All three should be opt-in flags on the `konrad run` subcommand (`--harden`, `--no-network`, `--ro-workspace`) so users who need looser access can still get it. Pairs with the Security audit todo.
- [ ] **Workspace snapshot / instant undo for agent runs.** Before launching the agent, snapshot the workspace so any destructive operation (mass delete, corrupt rewrite) can be undone with a single command. Implementation sketch: a `konrad snapshot` step (invoked automatically on `konrad run` unless `--no-snapshot`) creates a hidden `.konrad-undo/` directory using the fastest method available — `cp -Rc` (APFS clonefile) on macOS, `cp --reflink=auto` on Linux Btrfs/XFS (zero-space CoW, instant regardless of tree size), falling back to `cp -a` on Ext4/NTFS (still <0.2 s for a typical 100 MB project on NVMe). After the session, `konrad undo` swaps the folders back; `konrad commit-run` discards the snapshot. Advantages over git: survives a deleted `.git` dir, tracks untracked/binary files and `node_modules` without `.gitignore` negotiation, and is faster than `git add && git commit` for large binary assets. The snapshot dir should itself be added to `.gitignore` by `ensure_gitignore()` alongside `.agent/opencode/`.

## Implemented

- [x] **2026-05-25 — Reproducible layers via `SOURCE_DATE_EPOCH` + `rewrite-timestamp=true`.** Came out of a brainstorm about reducing user re-download size after the lock-by-digest work landed. **The problem:** even when CI rebuilds a layer that's *logically* identical to a previous run (same packages installed, same files copied), apt / npm / uv stamp the current build time into file mtimes. Tar archives include mtimes, so the layer's sha256 differs from the previous build, so users re-download bytes that are functionally identical. **The fix:** [.github/workflows/build-image.yml](.github/workflows/build-image.yml) now computes `COMMIT_EPOCH` (Unix timestamp of the commit being built) alongside the existing `COMMIT_DATE`, passes it as a `SOURCE_DATE_EPOCH` build-arg, and switches the build output from the bare `push: true` field to `outputs: type=image,push=true,rewrite-timestamp=true`. BuildKit then walks every file in every produced layer and rewrites the mtime to `SOURCE_DATE_EPOCH` before tar'ing. Two builds from the same source commit now produce byte-identical layers (modulo legitimate content changes); two builds from *different* commits whose layer-content happens to be byte-identical also produce byte-identical layers. **Effect on user pulls:** when a commit causes CI to rebuild a layer but its content is unchanged from the previous build (a common case after the LABEL-ordering fix landed earlier today), the rebuilt layer's diff_id now stays stable instead of getting a new digest. Users pull manifest metadata only. **Caveats.** Provenance attestations still include build-time timestamps; those are separate from layer digests so they don't affect caching, but they do mean the image isn't *strictly* bit-reproducible across runs in the SLSA sense — a known trade-off documented by the docker buildx team and acceptable for our cache-savings use case. **Surface inventory.** Edited: [.github/workflows/build-image.yml](.github/workflows/build-image.yml) only. No Dockerfile changes, no script changes, no behavioral changes inside the container. `bin/konrad rebuild` and `scripts/build-image.sh` intentionally NOT updated; local builds keep current-time mtimes for debugging convenience (the local-rebuild image lives under `konrad:local`, separate from the published `konrad:latest`, so there's no cache interaction).

- [x] **2026-05-25 — Cache-correctness fix in stage 3: move version LABEL + `COPY opencode/` past the tail.** Diagnosed from a scheduled CI build where stage-1 `apt-get install` hit `CACHED` but stage-3 didn't, despite `python.lock` and the base image digest both being unchanged — the structural asymmetry between the two stages was the smoking gun. **Root cause #1**: `ARG GIT_SHA` + LABEL using it sat between `FROM` and stage-3's `apt-get install`. GIT_SHA changes every commit, so the LABEL layer invalidated, cascading the cache miss through apt → fonts → typst → `COPY --from=python-base` → `COPY --from=python-models` → npm. Stage 1 escaped because it has no ARG-bearing LABEL between FROM and its apt step. **Root cause #2**: `COPY opencode/` sat downstream of the `build-manifest` RUN, so any commit (even a ROADMAP-only one) invalidated the manifest layer and cascaded into a fresh `COPY opencode/` whose tar content was identical to the previous run — a wasted re-pack and a new diff_id for users. **Fix**: all three commit-varying ARGs (`KONRAD_VERSION`, `GIT_SHA`, `BUILD_DATE`) and the version/revision LABEL are now declared together at the very tail of stage 3, *after* `COPY opencode/`. Order: ... → npm → `COPY opencode/` → ARGs + LABEL → build-manifest → USER → ENTRYPOINT. **Impact** validated end-to-end on the next CI run (commit 633ebeb, ROADMAP-only): stage-3 apt, fonts, typst, COPY --from layers, npm, and `COPY opencode/` all hit `CACHED`. Only the small LABEL + build-manifest tail rebuilt. CI build time dropped from ~10 min to ~2-3 min on no-op commits; user re-download for `:latest` dropped from ~600 MB to a few KB. Big layers (Docling models 1.1 GB, Python venv 447 MB, stage-3 apt 382 MB, npm 203 MB) now keep stable diff_ids across commits. No VERSION bump — in-container behavior is unchanged.

CI drops `no-cache: true` (kept `pull: true` on scheduled runs only, so Debian base-image point updates still flow in). Docker's GHA layer cache walks layers and only rebuilds the ones whose lock genuinely moved — typical "only opencode-ai bumped" day rebuilds only the npm layer and downstream (~80 MB user pull); no-op days are a cache hit top-to-bottom (~0 MB pull). **Daily refresh stays automatic.** New [.gitlab-ci.yml](.gitlab-ci.yml) `resolve-locks` job runs daily at 01:00 UTC on gitlab.git.nrw's shared docker runners (`tags: [docker]`) inside a `node:26-trixie-slim` container (same base as the runtime image, so Python resolution matches stage 1). The merge mirrors to GitHub and triggers `build-image.yml` via the `image/**` path filter; CI builds only the affected layers. Floating pins still flow in — just visibly and one input at a time. **Why GitLab, not GitHub Actions like the build pipeline:** the build needs Podman, which gitlab.git.nrw's shared runners can't grant (see 2026-05-23 entry below). Two builds from the same source commit now produce a byte-identical manifest layer — without this, the build-manifest layer would rehash daily (timestamp inside the JSON) and invalidate the small opencode COPY layer downstream of it. **Scope decisions worth recording.** *apt is not locked* — Debian doesn't have a clean "install from lock" format; per-package pinning is fragile because the archive rolls old versions; apt layer is small and downstream of nothing huge, so apt churn doesn't cost much. *uv is not locked* — uv installs from a fully-pinned lock reproducibly across versions, so uv-version churn doesn't matter for the resulting venv; keeping uv at `:latest` is simpler than locking it. *Models are not yet locked* — that's the (b) follow-up; same lock-file thinking applied to model weights via `models.lock`. **Surface inventory.** New files: [image/locks/python.spec](image/locks/python.spec), [image/locks/python.lock](image/locks/python.lock), [image/locks/npm.lock](image/locks/npm.lock), [image/locks/typst.lock](image/locks/typst.lock), [.gitlab-ci.yml](.gitlab-ci.yml).
