# Roadmap & Backlog

Tiers gate by release phase: **Tier 1** is the road to beta, **Tier 2** is
beta ‚Üí 1.0, **Tier 3** is post-1.0. Inside each tier, items cluster by theme.
Each item carries a rough t-shirt size ‚Äî `S` (hours), `M` (a day or two),
`L` (several days), `XL` (a week+) ‚Äî a planning guess, not a commitment.
Conventions for what lands here (vs. CHANGELOG or a commit message) are in
[CONTRIBUTING ‚Üí When to update what](CONTRIBUTING.md#when-to-update-what).

## Inbox

_Raw ideas land here. Promote into the appropriate tier after a refinement pass._

- [ ] **Confidential `context-confidential/` tier (opt-in mount).** A second tier of the `context/` mount (shipped) for sensitive material, enforced by mount _presence_, not an in-agent "ask": opencode has no read-permission primitive (only `edit`/`external_directory`/`bash`/`webfetch`/`websearch`), an `external_directory: ask` would gate only the Read tool and miss bash `grep`/`cat` (the natural access path), and a cooperation-dependent ask is the kind of boundary konrad refuses elsewhere (cf. the egress firewall). So host `~/.config/konrad/context-confidential/<name>/` ‚Üí guest `/context-confidential/<name>` `:ro`, mounted _only_ on opt-in (`--confidential` / `KONRAD_CONFIDENTIAL=1`) ‚Äî the "ask" moves to the human at launch, the only place it's a real boundary. Coarse boolean opt-in to start (the egress firewall already bounds exfil); per-name selection (`--context <name>`) only if all-or-nothing grants get too broad. Optional secondary speed-bump once mounted: `external_directory: {"/context-confidential/**": "ask"}` (Read-tool only ‚Äî polish, not the boundary). Caveat: only as private as the provider (the model sees whatever text it is handed), so pair with an internal endpoint. Builds on the shipped `context/` mount.

## Tier 1 ‚Äî Road to beta

_Gates the beta declaration ‚Äî a genuinely usable product for the organization._

- **Distribution & operations**
  - [ ] `M` **Auto-update.** Keep the latest image at hand ‚Äî possibly via a background refresh. Also allow e.g. org layers to ride with the Konrad auto-update to stay fresh (and so they do not need to deliver their own auto update mechanism) ‚Äî the org-layer half is the sibling **Org-layer subscription & lifecycle** item below (subscription sync already rides `konrad update`; once a background refresh lands, org layers ride it for free, no org-side updater).
  - [ ] `L` **Org-layer subscription & lifecycle (native `konrad org`).** Today an org maintains its layer with a bespoke host-side script it ships itself (fetch a ref, copy/`git archive` into `~/.config/konrad/org/`, mirror reference corpora into `context/`). One root cause ‚Äî konrad has no notion of "subscribe to and maintain an org layer" ‚Äî yields three smells: every org reinvents the same fetch/version/idempotency logic; the script lives _outside_ the artifact it maintains, so it's frozen at clone time; and host-side jobs (wiki mirror, per-member identity derivation) keep accreting onto a script konrad can't reason about. Make subscription a first-class, **git-native, declarative** concept and let orgs ship _only_ genuine deltas. **Subsumes the former "Multiple org layers, merged" item** (same merge engine), and is the mechanism behind the **Auto-update** item's "org layers ride the auto-update" half ‚Äî they ship together.
    - **Decided shape ‚Äî git-native subscriptions as managed mirrors.** `~/.config/konrad/org/` becomes a container of subscription subdirs, each a konrad-owned git checkout (`org/<name>/`). `konrad org add <git-url>` clones it; `konrad update` (and a manual `konrad org sync [<name>]`) re-syncs idempotently ‚Äî `git fetch && reset --hard` to a tracked **branch** (a moving HEAD, not a pinned tag: an org ships by pushing). Each dir is a **managed mirror** ‚Äî local edits in it are clobbered on sync; the user's override channel stays the `user/` layer, which still merges on top. **No state file**: the in-place `.git` records URL + branch, so `konrad org list` reads them straight from git config; `.git` rides harmlessly in the read-only mount.
    - **Multiple layers, merged (folded in).** Discover all `org/*/` and deep-merge in **alphabetical dir order** (last writer wins; control precedence with a numeric `--name` prefix, e.g. `10-`/`20-`) between baked and user ‚Äî `baked < org‚ÇÅ < ‚Ä¶ < user`. [image/merge-config.js](image/merge-config.js) is _already_ an N-input deep-merge, so this is entrypoint glob + mount wiring, not new merge code. Additive channels (`instructions/`/`skills/`/`agents/`/`fonts/`) accumulate per layer via opencode's own globbing ‚Äî widen the baked `instructions` glob `org/instructions/*.md` ‚Üí `org/*/instructions/*.md` (verify opencode's globber takes the extra wildcard level). The firewall's [compose-allowed-hosts.sh](image/konrad-defaults/compose-allowed-hosts.sh) must enumerate **every** `org/*/opencode.jsonc` when deriving provider hosts ‚Äî a missed layer silently blocks that org's internal provider.
    - **Versioned post-sync hook (the only org code).** For the ~10% needing real logic (derive operator identity from the org's own auth, e.g. a `glab`/`gh api` call; mirror a wiki) the layer ships `hooks/post-sync` _inside itself_ ‚Äî so it rides the branch and self-updates with everything else (a freestanding script konrad merely _points at_ would re-create the frozen-clone problem). Run **host-side, raw, pre-mount, on every sync**, cwd = the layer dir, **no special env** (the hook derives paths from the well-known `~/.config/konrad/` home ‚Äî the same "well-known `$HOME` folder, not an env var" principle as layer discovery); the hook owns its own idempotency. **Trust model: adding _is_ trusting** ‚Äî no prompt, no flag; subscribing to a layer is the decision to run its code (documented as the boundary). Context-mirroring stays **hook territory only** ‚Äî konrad gains no declarative corpus-sync engine; the hook writes into `~/.config/konrad/context/`.
    - **Auth by composition; `git` a soft dep.** konrad runs plain host `git` and ships **zero auth code** ‚Äî private internal repos work via the user's existing git credentials: a forge CLI login (`gh auth login` / `glab auth login`) does a browser OAuth flow _and_ installs a git credential helper, or SSH. `git` becomes a **feature-scoped soft dependency**: `install.sh` warns if absent, core konrad still runs, `konrad org` fails cleanly with "install git."
    - **Command surface.** `org add <url> [--name] [--branch]` (name from the repo basename, branch from the remote default, unless overridden; errors on an existing name), `org list` (bare `konrad org` too), `org remove <name>` (deletes the dir; does _not_ undo hook side effects like a `context/` mirror ‚Äî konrad can't know them), `org sync [<name>]` (no arg = all). **Sync failure = warn + keep last-good**: a flaky fetch leaves the prior checkout and lets `konrad update` finish (availability beats freshness for a defaults mechanism).
    - **Breaking change (acceptable pre-1.0).** A bare `~/.config/konrad/org/` holding config _directly_ (today's copy-a-payload model) stops being a layer ‚Äî everything must live in `org/<name>/`. Ships with a one-line `mv` migration in [CHANGELOG.md](CHANGELOG.md) + upgrade notes; [`examples/org-package/`](examples/org-package/) drops `install.sh` and becomes a git-native example repo (sample config + `hooks/post-sync`). Touches `image/` (entrypoint + globs) so it fires a rebuild; minor `VERSION` bump; README "For organizations" rewrite + `git` requirement; ARCHITECTURE note (org layer goes from "files in your home" to "managed git mirrors"; the hook is a deliberate exception to "agent code runs only in the sandbox").
    - **Open before shipping.** Must every `org/<name>/` be git-backed, or is a manual non-subscribed named layer (`org/local/`, no `.git`) also a valid merge input (it just can't `sync`)? Confirm opencode's instruction/skill globber resolves the added wildcard level. Proposed defaults to confirm: shallow `--depth 1 --single-branch` clones; hook stdout streams to the user.
- **Security & sandboxing**
  - [ ] `L` **Security audit.** End-to-end review before declaring beta: container isolation, provider-credential handling, MCP tool surface, filesystem-access boundaries...
    - is the container hardened enough against common threats (e.g. via pids-limit?)
  - [ ] `?` **Investigate: `egress firewall: could not create internal network` on Apple `container`.** Seen 2026-06-23 on macOS (`konrad` 0.17.0): every firewall-on launch failed at net creation (each after a Ctrl-C of an apparently-hung `container network create --internal`); `--no-firewall` launches worked. The machine also held 8 orphaned `konrad-fw-net-<pid>` nets + a running orphan `konrad-fw-proxy-8405`, and a `container system stop && start` was what cleared it at the time. **Pool-exhaustion theory falsified (2026-06-24):** 41 deliberately-leaked `konrad-fw-net-*` orphans created cleanly (subnets through `192.168.106.0/24`) and the unfixed `konrad` then launched fine with all 41 present ‚Äî so orphan accumulation neither exhausts the vmnet pool nor causes the wedge. Leading hypothesis now: a transient `container-apiserver` state (hence the stop/start fix), not anything konrad does. Next time it recurs, **capture a live repro** ‚Äî `container system logs`, daemon state, and whether a bare `container network create --internal foo` hangs _without_ konrad in the loop ‚Äî before changing code. (A startup-reaper + interrupt-teardown fix was built and reverted as speculative once the theory was falsified; don't rebuild it without a repro.) The orphaned-leftover cleanup is still mildly worth doing, but as the on-demand `reset`/`uninstall` sweep under the Tier-2 dev-container item ‚Äî not an automatic per-launch reaper.

- **Dev Container (second consumption path)**
  - [ ] `XL` **`konrad init-devcontainer`.** Scaffold a `.devcontainer/` into the _user's_ project, so they drive Konrad from VS Code instead of the terminal. Ships as a new subcommand ‚Üí minor `VERSION` bump + README "Dev Container" section + CHANGELOG; deletes on ship. Not Konrad's own `.devcontainer/` (that configures working _on_ Konrad).
    - **Decided shape:** the published Konrad image _is_ the dev container, sealed on an `--internal` Podman network with the same egress-firewall sidecar the CLI uses ‚Äî not image-as-toolbox (can't host the firewall ‚Äî the proxy needs its own netns) and not a socket-mounting re-run of the host CLI (drags in the remote-daemon apparatus for no gain). Used exactly like the CLI (pure agent, no in-container dev work), so it needs no dev egress and the firewall floor stays as tight as the CLI's.
    - **Validated by probe (2026-06-12):** `konrad-entrypoint <anycmd>` composes `baked < org < user` independently of what it execs; an image container on a `--internal` net is fully sealed (DNS fails); the proxy sidecar started _outside_ `podman_run` filters allowed ‚Üí 200 / denied ‚Üí 403 ‚Äî i.e. the firewall assembles by hand exactly as the CLI assembles it.
    - **DRY constraint ‚Äî reuse, never reimplement:** config compose via `postStartCommand: konrad-entrypoint true`; agent + proxy are the same baked image; extract the network+proxy create/teardown from `fw_setup`/`fw_teardown` into shared helpers used by _both_ the per-run CLI path and a new persistent `konrad fw up/down <id>` the dev container invokes; generate `devcontainer.json` from `bin/konrad`'s existing constants (one source of truth); credentials + model carry via the shared `konrad-secrets` / `-state` / `-cache` volumes. Template is heredoc(s) in `bin/konrad`; image line stamped with the running CLI's `:0.X.Y`.
    - **Open before shipping:** does VS Code provision `vscode-server` host-side (zero container egress ‚Äî the expected behavior that makes the tight firewall viable) or insist on a container-side fetch (then a single devcontainer-scoped `KONRAD_ALLOWED_HOSTS` entry for just the VS Code host, never the dev set)? Make `konrad fw up` idempotent + self-cleaning; have `reset` / `uninstall` sweep stray `konrad-fw-*`.

## Blocked (upstream)

_Items that depend on an upstream fix where we've chosen not to ship a workaround. Revisit periodically._

- [ ] `M` **`preserve_thinking` is a no-op through opencode.** Cross-turn reasoning retention does not work on the tested OpenAI-compatible path. Investigated 2026-06-12 (uzk-gpt endpoint / Qwen3.6-27B, full round trip): the endpoint honors `preserve_thinking` only for inline `<think>` in assistant content; it ignores history thinking sent as a separate `reasoning_content` field ‚Äî the _only_ form opencode emits (it forwards the flag and resends prior reasoning, but never re-inlines `<think>`). So the flag is harmless but does nothing. Decision needed: **(a)** endpoint-side ‚Äî configure the served chat template / reasoning parser to reconstruct `<think>` from input `reasoning_content` (helps every OpenAI-compatible client); **(b)** opencode-side ‚Äî re-inline `<think>` into history (needs an upstream change or a plugin); or **(c)** document as a known limitation and drop `preserve_thinking` from the recommended config. Touches the "Local models" shakedown and the A/B-test item's reproducibility needs.
- [ ] `S` **Image build date shows 1970 (`config.created`).** Reproducible builds set `SOURCE_DATE_EPOCH=0`, so `podman images` / `docker inspect .Created` report the epoch. Verified (2026-06) that `config.created` can't be decoupled from the layer-rewrite epoch: dropping `SOURCE_DATE_EPOCH` re-stamps every layer to build time (breaks dedup), and the exporter's `source-date-epoch` attribute overrides `config.created` too ‚Äî both knobs set one shared value. Only a post-build mutation of the config blob (layers left byte-identical) could give a real date ‚Äî rejected as not worth the bespoke multi-arch tooling for a cosmetic field. The real build time still lives in `/etc/konrad/build-manifest.json` and the OCI labels. Revisit if the build tooling gains a way to set `config.created` independently of the layer epoch.
