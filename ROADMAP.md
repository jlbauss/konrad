- [ ] `XL` **`konrad init-devcontainer` (second consumption path).** Scaffold a `.devcontainer/` into the _user's_ project, so they drive Konrad from VS Code instead of the terminal. Ships as a new subcommand → minor `VERSION` bump + README "Dev Container" section + CHANGELOG; deletes on ship. Not Konrad's own `.devcontainer/` (that configures working _on_ Konrad).
  - **Decided shape:** the published Konrad image _is_ the dev container, sealed on an `--internal` Podman network with the same egress-firewall sidecar the CLI uses — not image-as-toolbox (can't host the firewall — the proxy needs its own netns) and not a socket-mounting re-run of the host CLI (drags in the remote-daemon apparatus for no gain). Used exactly like the CLI (pure agent, no in-container dev work), so it needs no dev egress and the firewall floor stays as tight as the CLI's.
  - **Validated by probe (2026-06-12):** `konrad-entrypoint <anycmd>` composes `baked < org < user` independently of what it execs; an image container on a `--internal` net is fully sealed (DNS fails); the proxy sidecar started _outside_ `podman_run` filters allowed → 200 / denied → 403 — i.e. the firewall assembles by hand exactly as the CLI assembles it.
  - **DRY constraint — reuse, never reimplement:** config compose via `postStartCommand: konrad-entrypoint true`; agent + proxy are the same baked image; extract the network+proxy create/teardown from `fw_setup`/`fw_teardown` into shared helpers used by _both_ the per-run CLI path and a new persistent `konrad fw up/down <id>` the dev container invokes; generate `devcontainer.json` from `bin/konrad`'s existing constants (one source of truth); credentials + model carry via the shared `konrad-secrets` / `-state` / `-cache` volumes. Template is heredoc(s) in `bin/konrad`; image line stamped with the running CLI's `:0.X.Y`.
  - **Open before shipping:** does VS Code provision `vscode-server` host-side (zero container egress — the expected behavior that makes the tight firewall viable) or insist on a container-side fetch (then a single devcontainer-scoped `KONRAD_ALLOWED_HOSTS` entry for just the VS Code host, never the dev set)? Make `konrad fw up` idempotent + self-cleaning; have `reset` / `uninstall` sweep stray `konrad-fw-*`.
- [ ] `M` **Confidential `context-confidential/` tier (opt-in mount).** A second tier of the `context/` mount (shipped) for sensitive material, enforced by mount _presence_, not an in-agent "ask": opencode has no read-permission primitive (only `edit`/`external_directory`/`bash`/`webfetch`/`websearch`), an `external_directory: ask` would gate only the Read tool and miss bash `grep`/`cat` (the natural access path), and a cooperation-dependent ask is the kind of boundary konrad refuses elsewhere (cf. the egress firewall). So host `~/.config/konrad/context-confidential/<name>/` → guest `/context-confidential/<name>` `:ro`, mounted _only_ on opt-in (`--confidential` / `KONRAD_CONFIDENTIAL=1`) — the "ask" moves to the human at launch, the only place it's a real boundary. Coarse boolean opt-in to start (the egress firewall already bounds exfil); per-name selection (`--context <name>`) only if all-or-nothing grants get too broad. Optional secondary speed-bump once mounted: `external_directory: {"/context-confidential/**": "ask"}` (Read-tool only — polish, not the boundary). Caveat: only as private as the provider (the model sees whatever text it is handed), so pair with an internal endpoint. Builds on the shipped `context/` mount.

## Blocked (upstream)

_Items that depend on an upstream fix where we've chosen not to ship a workaround. Revisit periodically._

- [ ] `M` **`preserve_thinking` is a no-op through opencode — blocked on a reasoning field-name mismatch.** By design, `preserve_thinking` is a Qwen chat-template kwarg: the client passes prior reasoning back as a _field_ on history assistant messages and the server template re-inlines it as `<think>` — so opencode's behavior (forward the kwarg, resend reasoning as `reasoning_content`) is correct-by-design. The blocker: vLLM ≥ 0.20 renamed the field to `reasoning` (RFC vllm#27755) and its input parser silently drops incoming `reasoning_content` (vllm#38488, open) — exactly matching the 2026-06-12 uzk-gpt/Qwen3.6-27B test (inline `<think>` honored, `reasoning_content` ignored). Unblocks when either **(a)** the endpoint's vLLM gains the input fallback (vllm#38488), or **(b)** opencode/AI-SDK emits `reasoning` (vercel/ai#6226, opencode#35283/#35284; qwen-code already shipped this client-side fix in its PR #4294) — else **(c)** document as a known limitation and drop the flag from the recommended config. Confirming probe (curl-only, next uzk-gpt access): replay history with reasoning under `reasoning` + `chat_template_kwargs.preserve_thinking: true`, with inline-`<think>` (known good) and `reasoning_content` (known bad) as controls. Changelog re-checked 2026-07-15 (v1.17.3 → v1.18.1): nothing relevant. Touches the "Local-model end-to-end shakedown" and A/B-test items' reproducibility needs.
- [ ] `S` **Image build date shows 1970 (`config.created`).** Reproducible builds set `SOURCE_DATE_EPOCH=0`, so `podman images` / `docker inspect .Created` report the epoch. Verified (2026-06) that `config.created` can't be decoupled from the layer-rewrite epoch: dropping `SOURCE_DATE_EPOCH` re-stamps every layer to build time (breaks dedup), and the exporter's `source-date-epoch` attribute overrides `config.created` too — both knobs set one shared value. Only a post-build mutation of the config blob (layers left byte-identical) could give a real date — rejected as not worth the bespoke multi-arch tooling for a cosmetic field. The real build time still lives in `/etc/konrad/build-manifest.json` and the OCI labels. Revisit if the build tooling gains a way to set `config.created` independently of the layer epoch.
