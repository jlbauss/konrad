This is the model-layer-specific follow-up. **The fake dependency.** The ~1.1 GB Docling model layer at stage 2 (`python-models` in [image/Dockerfile](image/Dockerfile)) inherits from `python-base`, so any `python.lock` bump invalidates the model layer in CI. The rebuilt layer gets a new diff_id even when the weights are byte-identical to what users already have — different file mtimes from the re-download, plus HuggingFace cache metadata that varies per run. Result: users re-download 1.1 GB every time a Python dep bumps, even though docling's required weights didn't change. The cache key dependency is structural, not semantic — the weights don't actually depend on torch's version. **The fix.** Restructure stage 2 to be standalone — `FROM ${BASE_IMAGE}` (not `python-base`), `apt install python3 ca-certificates`, `pip install huggingface_hub`, then a Python fetcher that calls `huggingface_hub.snapshot_download(repo_id, revision=<sha>)` for each repo in a new `image/locks/models.lock`. After download, `rm -rf /opt/models/.cache /opt/models/.locks` strips non-content HF metadata. Combined with the already-landed `SOURCE_DATE_EPOCH` + `rewrite-timestamp` flow, mtimes are normalized at layer-production time, so what's left is byte-stable across rebuilds. Cache key for the whole stage = `models.lock` contents only — independent of `python.lock`, `base.lock`, anything else. Users skip the 1.1 GB whenever the lock didn't move (the common case for python.lock bumps that touch pypdf / pdfplumber / onnxruntime without moving docling's required model set). **`models.lock` format**: one line per repo, `<hf_repo_id> sha256:<commit>`. **GitLab bot extension**: a sixth resolver block in [.gitlab-ci.yml](.gitlab-ci.yml) that hits `https://huggingface.co/api/models/<repo>/revision/main` per repo in the lock and bumps if the SHA moved — same diff-and-merge plumbing as the other five locks. **Model-set changes** (docling deciding to use a new repo entirely on a version bump) need manual maintenance: the bot only knows about repos already in the lock, so a new docling release that depends on a 6th repo would fail at build or runtime, and a maintainer edits `models.lock` by hand to add the new entry. The earlier sketch of this bullet referenced a `docling-tools models download --dry-run` flag for auto-detecting model-set changes — that flag does not exist (verified 2026-05-25 by reading docling's CLI source); the bot's silence on new repos is the safety net, not an active detector. **BuildKit cache mount** on the fetcher RUN (`--mount=type=cache,target=/root/.cache/huggingface`) so CI re-fetches over the network only what genuinely moved. **Research checklist before implementation** (one CI run on a machine with Python + docling-slim of the version pinned in [image/locks/python.lock](image/locks/python.lock)): (a) exact list of HF repo IDs `docling-tools models download` resolves for our pinned docling-slim version — enumerate by reading docling's source under `docling/datamodel/` and `docling/utils/model_downloader.py`, or by running and `find /opt/models -name '*.safetensors'`; (b) whether `snapshot_download(repo, revision=<sha>)` produces a byte-stable directory tree across two runs after stripping `.cache/` and `.locks/` — `diff -r run1 run2` should be empty; (c) whether docling's runtime resolves models via the HuggingFace cache convention (`models--<org>--<repo>/snapshots/<sha>/`) or via absolute paths — determines whether the fetcher needs to mimic the HF directory layout or can dump weights to flat paths. **Alternative considered: models in a Podman volume** (the "2a" option from the 2026-05-25 brainstorm). Removes 1.1 GB from the image entirely; users download once and reuse across `konrad update` operations. Trade-off: breaks the "konrad works offline immediately after pull" property — first run becomes "wait 5-10 min for the model fetch." Mitigations (pre-warm during `konrad install`, opt-in via env var) exist but each adds complexity. Treated as a lower-priority fallback: if the lock approach above proves too fragile in practice (e.g., HF revisions disappear, snapshot_download output won't normalize, docling's runtime path-resolution is too brittle), or if image size becomes a primary user complaint, this is the natural next step.

## Implemented

### Tier 1 — alpha → beta

- [x] **2026-05-29 — Three-segment pre-1.0 versions (`0.X.Y`) + hyphen-separated dated tags.** Reworked the pre-1.0 versioning rule so `X` (minor) and `Y` (patch) carry distinct meaning instead of `X` inflating on every functional change. **The rule.** Bump `Y` for a bug fix with no surface change (same flags, same outputs, same behavior on the happy path); bump `X` for new functionality, behavior change, base-image major bump, or anything a user might notice (reset `Y` to 0). No MAJOR slot pre-1.0 — would-be-major changes land as minor under the leading `0.`. Doc-only / CI-only / ROADMAP-only changes still don't bump. Post-1.0 stays full semver, unchanged. **Why.** The previous "bump `X` for any functional change" rule made `X` inflate fast enough that the minor lifetime measured in hours, not features — `X` had stopped signalling anything to users. Splitting fixes (patch) from features (minor) at the version level matches what npm/cargo already assume for `0.x.y`, and gives users a coarse "quiet refresh vs changed something" signal without forcing a 1.0 declaration before we're ready. **Tag-scheme touch-up.** The dated immutable tag's separator flipped from dot to hyphen (`:0.2.YYYY-MM-DD` → `:0.2.1-YYYY-MM-DD`) — a dot would have made the tag read as a four-segment version. New `:0.X.Y` rolling tag tracks the patch line; existing `:0.X` rolling tag now resolves through the new patch slot to mean "latest patch on the `0.X` line." `:latest` and `:<sha>` unchanged. CLI's `image_dated_tag()` in [bin/konrad](bin/konrad) tracks the same separator so `konrad --version` and the registry agree. **Surface inventory.** Edited: [CLAUDE.md](CLAUDE.md) (versioning working-agreement bullet + trunk-flow tag scheme parenthetical), [docs/design/versioning-and-releases.md](docs/design/versioning-and-releases.md) (Pre-1.0 section, pre-1.0 tag table, examples table, day-zero history entry for `0.2.1`, intro example), [.github/workflows/build-image.yml](.github/workflows/build-image.yml) (publish step generates new tag list via `${VER%.*}` for the minor float; header comments updated), [bin/konrad](bin/konrad) (`image_dated_tag` dot → hyphen). No `:0.2.0` tag was ever published — the line skips straight from `:0.2.<date>` (old format, last seen 2026-05-29) to `:0.2.1-<date>` (new format) at the transition.
