- [ ] **Dedupe the two local rebuild paths.** `bin/konrad`'s `do_rebuild()` and [scripts/build-image.sh](scripts/build-image.sh) are near-duplicate three-stage `podman build` flows (lock reading, git-sha/build-date metadata, the same build-args) — and they already drifted once (the 0.6.0 image-dedupe work touched one before the other). - [ ] **Split the heavy Python deps into their own layer (venv pull-dedupe).** Sibling to the 0.6.0 model-layer dedupe. The `COPY /opt/venv` layer is ~1.3 GB — now the biggest layer in the image — and re-pulls on every `python.lock` bump (the most frequent bump, since the daily bot moves the lock for any transitive dep). Most of that mass is torch/torchvision (CPU build, a few hundred MB) plus a few other large, rarely-moving wheels (opencv, scipy, numpy); a trivial `pypdf` bump drags all of torch with it. **Idea:** install the venv in two phases — the heavy/stable set (torch, torchvision, …) vs. everything else — laid down as separate image layers, so a small-dep bump re-pulls only the small layer. **Fiddly part:** site-packages intermixes packages, so the split needs care (two `uv pip install` passes with the heavy set COPY'd as its own layer by package dir + `.dist-info`, or a constraints split) and the heavy layer must stay byte-stable under `rewrite-timestamp`. Decide the split empirically from `podman history` / wheel sizes — torch + torchvision are the obvious anchors. Surfaced 2026-06-09 from 0.6.0's layer breakdown.

## Tier 2 — beta → 1.0

_Work done after the repo goes public as beta. Mostly speculative or aspirational._

- [ ] **Add automated test suite** and sensible and fast way to do test runs, before commiting to main
- [ ] **A/B-test prompts and skills by letting a frontier model drive konrad.** Evolve the baked skills and agent prompts empirically instead of by hand: run konrad end-to-end against a fixed task twice — once with skill/prompt variant A, once with B — and judge which output is better. **Depends on the non-interactive `konrad run` mode + isolated self-test profile (landed 2026-06-08 in 0.5.0 — see CHANGELOG.md)** — that's the primitive the whole harness is built on. **Substrate sketch:** baseline-vs-variant images (or a runtime skill-dir overlay so a variant swaps in without a full rebuild); the same fixed prompt against each; a pinned cheap/fast model so cost stays bounded and results aren't confounded by model drift (the CLAUDE.md self-testing rule pins `openrouter/qwen/qwen3.6-27b` for end-to-end runs); fixed temperature/seed where the model allows, for reproducibility; captured structured output diffed or scored by a judge model; isolated `konrad-selftest-*` volumes per run so every comparison starts from clean state.
- [ ] **konrad --update progress UX: stop fighting terminals; switch to append-only progress with OS-aware fallback.** Findings from Podman show macOS (podman machine remote mode) cannot reliably render TTY progress bars and degrades to Copying blob sha256: output, while Linux retains full native progress UI. SSH-ing into the VM is not a dependable UX path for install workflows. Decision: abandon redraw-based progress (\r, ANSI, cursor tricks) and introduce a simple OS conditional layer. Implementation (if/else):
if macOS: use custom append-only progress (left-to-right bar, one immutable step per layer/event, no redraw assumptions)
else (Linux): call podman pull directly and preserve native progress output
Progress model (macOS):
Header: Pulling konrad image — N layers
Progress: append-only blocks (██████…) per layer/event
Final line: ✓ updated → image x.y.z (N layers)
No terminal detection beyond OS switch, no CR redraws, no ANSI cursor movement, no /dev/tty hacks. Optional KONRAD_PROGRESS=overwrite retains legacy behavior.

The CLI is the first surface every user touches; it should feel cared-for.
- [ ] **Mirror published images to the GitLab Container Registry (single-home distribution).** Today images publish only to `ghcr.io/jlbauss/konrad` (GitHub Actions builds them; the GitHub side exists purely as a CI execution surface). Mirroring them to GitLab's registry makes `gitlab.git.nrw` the single home for everything a user touches — repo, `curl|sh` installer (already GitLab raw), images, and releases — and removes the user flow's dependency on GitHub/GHCR being reachable. Fits the sovereignty story (gitlab.git.nrw is first-party infra). **Prerequisites confirmed 2026-05-27**: (a) Container Registry is enabled for the project; (b) **anonymous pulls are allowed** — verified by requesting a token from `https://gitlab.git.nrw/jwt/auth?service=container_registry&scope=repository:jbauss2/konrad:pull` with no credentials and getting back `access: [{name: jbauss2/konrad, actions: [pull]}]` (registry host `registry.git.nrw`, realm `gitlab.git.nrw/jwt/auth`, project_id 3917). So a logged-out `podman pull registry.git.nrw/jbauss2/konrad:<tag>` will work once images land there. **Implementation sketch.** Don't build on GitLab — still blocked by the privileged-Podman / shared-runner constraint, so the build stays GitHub Actions → GHCR. Instead add one `skopeo copy --all` (or `crane cp`) step at the tail of [build-image.yml](.github/workflows/build-image.yml)'s publish job that mirrors each published tag (`:0.X.<date>`, `:0.X`, `:latest`, `:<sha>`) GHCR → `registry.git.nrw/jbauss2/konrad`. Multi-arch manifest copy, unprivileged, ~seconds. **Do the copy from GitHub Actions** (where the just-built image is already in hand) rather than a GitLab-side job — a GitLab job fires on push *before* GitHub finishes building, so it would race the not-yet-published GHCR image. The only new maintenance surface is one secret: a **GitLab project deploy token** with `write_registry` scope, stored as a GitHub Actions secret (skopeo authenticates to `registry.git.nrw` with it). **CLI change**: point `REGISTRY_IMAGE` in [bin/konrad](bin/konrad) at `registry.git.nrw/jbauss2/konrad:latest` as the default pull source (or accept both with GitLab as default); the installer already fetches the CLI from GitLab raw, so this closes the loop. **Keep GHCR too** — GitHub Actions needs it as the build target regardless, and it stays a fine fallback/override via `KONRAD_IMAGE`; GitLab just becomes the user-facing default. Functional change to `bin/konrad` + CI → bumps `VERSION`. Deserves its own commit + a real CI round-trip (can't be smoke-tested without podman + the publish pipeline). Pairs with the Homebrew-tap and `curl|sh` installer story as the third leg of "everything a user needs is on GitLab."
- [ ] **Collapse the two-forge setup if a single capable runner appears (standing simplification trigger).** The GitLab-primary / GitHub-mirror split exists for exactly one external reason: gitlab.git.nrw's shared runners can't run the privileged Podman build (see the [build-image.yml](.github/workflows/build-image.yml) header and the 0.1 entry in CHANGELOG.md, 2026-05-23). That one constraint radiates outward into most of the project's *governance* complexity — a one-way mirror to keep in sync, a "repo here, CI there" mental model, and a three-way contributor-flow fork in [CONTRIBUTING.md](CONTRIBUTING.md) (maintainer ff-only / collaborator MR-or-handoff / external fork-MR-with-no-auto-build). **The lever:** the day a single runner that *can* run the privileged build becomes available — a self-hosted GitLab runner, or gitlab.git.nrw offering one — the GitHub mirror can be deleted and the whole flow collapses to "branch, push, merge" on one forge, with the contributor tiers folding into one. The point of recording it here is to keep the mirror from calcifying into assumed-permanent infrastructure when it's actually *contingent*: revisit whenever the runner situation changes. Distinct from (and supersedes) the GitLab-registry-mirror item above — that one keeps GitHub-as-builder and mirrors the *artifacts* to GitLab; this one removes GitHub-as-builder entirely, so if this lands, that item is moot.
- [ ] **Homebrew tap as a second install path.** The `curl … | sh` installer landed 2026-05-26 and is the recommended path; a Homebrew tap is the natural complement for the brew-flavoured corner of the user base. Defer until both land.
- [ ] **Workspace snapshot / instant undo for agent runs.** Before launching the agent, snapshot the workspace so any destructive operation (mass delete, corrupt rewrite) can be undone with a single command. Implementation sketch: a `konrad snapshot` step (invoked automatically on `konrad run` unless `--no-snapshot`) creates a hidden `.konrad-undo/` directory using the fastest method available — `cp -Rc` (APFS clonefile) on macOS, `cp --reflink=auto` on Linux Btrfs/XFS (zero-space CoW, instant regardless of tree size), falling back to `cp -a` on Ext4/NTFS (still <0.2 s for a typical 100 MB project on NVMe). After the session, `konrad undo` swaps the folders back; `konrad commit-run` discards the snapshot. Advantages over git: survives a deleted `.git` dir, tracks untracked/binary files and `node_modules` without `.gitignore` negotiation, and is faster than `git add && git commit` for large binary assets. The snapshot dir should itself be added to `.gitignore` by `ensure_gitignore()` alongside `.agent/quality-assurance/` and `.agent/scratch/`.

- [ ] **PDF region discovery — additional helpers as real tasks demand.** Day-1 helpers (`find_words`, `find_lines` in `scripts/pdf_helpers.py`) cover word-level and line-level discovery via predicate; the strategies catalog in `annotate.md` walks how to combine them for phrase / paragraph / section / value-after-label / PII matching. (4) **`find_table_cells`** — typed table-cell discovery; pdfplumber's `find_tables()` exists but the cell-bbox shape isn't pleasant to use raw. Ship each only when a real user task demands it — speculative day-2 helpers tend to drift before they're proven.
