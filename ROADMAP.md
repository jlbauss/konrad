- [ ] **Cleanliness / simplification pass — pay down accumulated complexity.** A periodic, deliberate sweep to keep Konrad simple, logical, and smooth as it grows (the maintainer's standing ideal — see [ARCHITECTURE.md → Engineering ethos](ARCHITECTURE.md#engineering-ethos--lightweight-composed-low-maintenance) and the CLAUDE.md "keep it simple" working agreement). Not a single feature: a recurring "step back and tidy" pass that hunts for workarounds that should have been structural fixes, mechanisms that grew more clever than they need to be, near-duplicate logic across `bin/konrad` / `entrypoint.sh`, doc sprawl, and surprising behaviour that violates least-astonishment. **First concrete entry (surfaced 2026-06-02):** the `node` user doesn't own its whole `$HOME` — `install -d` leaves auto-created intermediate dirs root-owned, so the image works around it by naming each scaffolding dir, and a clean `chown -R /home/node` is blocked only by the 1.1 GB model cache living in the image. The structural fix is to move the model cache out of the image (see the Tier-1 image-dedupe "2a" option), after which node can simply own its home. Treat this bullet as the home for "this felt illogical, fix it properly later" notes so they don't get lost as one-off patches; promote individual items out when they're worth doing on their own. Also here: evaluate if it is useful to restructure the working directory order
- [ ] **Image-pull dedupe (b) — decouple the Docling model layer from the Python venv via a model lock.** Part (a) landed 2026-05-24 (see CHANGELOG.md / git history); foundations also extended on 2026-05-25 (lock-by-digest infrastructure + `SOURCE_DATE_EPOCH` layer normalization). This is the model-layer-specific follow-up. **The fake dependency.** The ~1.1 GB Docling model layer at stage 2 (`python-models` in [image/Dockerfile](image/Dockerfile)) inherits from `python-base`, so any `python.lock` bump invalidates the model layer in CI. The rebuilt layer gets a new diff_id even when the weights are byte-identical to what users already have — different file mtimes from the re-download, plus HuggingFace cache metadata that varies per run. Result: users re-download 1.1 GB every time a Python dep bumps, even though docling's required weights didn't change. The cache key dependency is structural, not semantic — the weights don't actually depend on torch's version. **The fix.** Restructure stage 2 to be standalone — `FROM ${BASE_IMAGE}` (not `python-base`), `apt install python3 ca-certificates`, `pip install huggingface_hub`, then a Python fetcher that calls `huggingface_hub.snapshot_download(repo_id, revision=<sha>)` for each repo in a new `image/locks/models.lock`. After download, `rm -rf /opt/models/.cache /opt/models/.locks` strips non-content HF metadata. Combined with the already-landed `SOURCE_DATE_EPOCH` + `rewrite-timestamp` flow, mtimes are normalized at layer-production time, so what's left is byte-stable across rebuilds. Cache key for the whole stage = `models.lock` contents only — independent of `python.lock`, `base.lock`, anything else. Users skip the 1.1 GB whenever the lock didn't move (the common case for python.lock bumps that touch pypdf / pdfplumber / onnxruntime without moving docling's required model set). **`models.lock` format**: one line per repo, `<hf_repo_id> sha256:<commit>`. **GitLab bot extension**: a sixth resolver block in [.gitlab-ci.yml](.gitlab-ci.yml) that hits `https://huggingface.co/api/models/<repo>/revision/main` per repo in the lock and bumps if the SHA moved — same diff-and-merge plumbing as the other five locks. **Model-set changes** (docling deciding to use a new repo entirely on a version bump) need manual maintenance: the bot only knows about repos already in the lock, so a new docling release that depends on a 6th repo would fail at build or runtime, and a maintainer edits `models.lock` by hand to add the new entry. The earlier sketch of this bullet referenced a `docling-tools models download --dry-run` flag for auto-detecting model-set changes — that flag does not exist (verified 2026-05-25 by reading docling's CLI source); the bot's silence on new repos is the safety net, not an active detector. **BuildKit cache mount** on the fetcher RUN (`--mount=type=cache,target=/root/.cache/huggingface`) so CI re-fetches over the network only what genuinely moved. **Research checklist before implementation** (one CI run on a machine with Python + docling-slim of the version pinned in [image/locks/python.lock](image/locks/python.lock)): (a) exact list of HF repo IDs `docling-tools models download` resolves for our pinned docling-slim version — enumerate by reading docling's source under `docling/datamodel/` and `docling/utils/model_downloader.py`, or by running and `find /opt/models -name '*.safetensors'`; (b) whether `snapshot_download(repo, revision=<sha>)` produces a byte-stable directory tree across two runs after stripping `.cache/` and `.locks/` — `diff -r run1 run2` should be empty; (c) whether docling's runtime resolves models via the HuggingFace cache convention (`models--<org>--<repo>/snapshots/<sha>/`) or via absolute paths — determines whether the fetcher needs to mimic the HF directory layout or can dump weights to flat paths. **Alternative considered: models in a Podman volume** (the "2a" option from the 2026-05-25 brainstorm). Removes 1.1 GB from the image entirely; users download once and reuse across `konrad update` operations. Trade-off: breaks the "konrad works offline immediately after pull" property — first run becomes "wait 5-10 min for the model fetch." Mitigations (pre-warm during `konrad install`, opt-in via env var) exist but each adds complexity. Treated as a lower-priority fallback: if the lock approach above proves too fragile in practice (e.g., HF revisions disappear, snapshot_download output won't normalize, docling's runtime path-resolution is too brittle), or if image size becomes a primary user complaint, this is the natural next step. **New argument for 2a — simplicity / "node owns its whole home" (surfaced 2026-06-02 during the org-config-layer font overlay).** The font overlay needed to `mkdir ~/.local/share/fonts` and hit a permission denial: `install -d -o node -g node …/.local/share/opencode` only chowns the dirs it's *named*, so auto-created intermediate parents (`.local`, `.local/share`, `.cache`) default to **root-owned**, and the runtime `node` user can't create new dirs under them. The obvious fix — `chown -R node:node /home/node` — is exactly what we *can't* do cheaply **because the 1.1 GB model cache lives in the image**: overlayfs copies a file up on any metadata change, so a recursive chown would duplicate the whole cache into a new layer (~doubling it). We worked around it by naming each intermediate scaffolding dir in `install -d` (owning empty dir inodes, no recursion) — but that's whack-a-mole, and it *feels illogical that the agent (node) doesn't simply have write rights everywhere in its own home*. This is a latent footgun: every future feature that writes somewhere new under `$HOME` risks the same surprising denial, and the maintainer's stated preference is "simple, logical, smooth" over clever-but-fragile. **Moving the models out of the image (2a) dissolves this entire class of problem**: with no heavy cache baked in, `chown -R node:node /home/node` (or just owning home outright) becomes free, node owns its whole home, and the scaffolding-naming dance goes away. So re-weight 2a — the win isn't only pull-dedupe and image size; it's also collapsing the home-ownership complexity into the one-liner it should be. Weigh that against the offline-first first-run cost when this is picked up.

- [ ] **konrad --update progress UX: stop fighting terminals; switch to append-only progress with OS-aware fallback.** Findings from Podman show macOS (podman machine remote mode) cannot reliably render TTY progress bars and degrades to Copying blob sha256: output, while Linux retains full native progress UI. SSH-ing into the VM is not a dependable UX path for install workflows. Decision: abandon redraw-based progress (\r, ANSI, cursor tricks) and introduce a simple OS conditional layer. Implementation (if/else):
if macOS: use custom append-only progress (left-to-right bar, one immutable step per layer/event, no redraw assumptions)
else (Linux): call podman pull directly and preserve native progress output
Progress model (macOS):
Header: Pulling konrad image — N layers
Progress: append-only blocks (██████…) per layer/event
Final line: ✓ updated → image x.y.z (N layers)
No terminal detection beyond OS switch, no CR redraws, no ANSI cursor movement, no /dev/tty hacks. Optional KONRAD_PROGRESS=overwrite retains legacy behavior.
- [ ] **Self-testing on macOS — needs a VM-side TCP/proxy daemon (deferred; clean socket route is a dead end).** Same shape as the progress-UX item above: works on Linux, blocked on macOS, parked. The OS-optional mount landed 2026-06-05 (see git history); the container *starts* on macOS, but self-testing is deliberately **off** there (`ensure-podman-sock.sh` → `/dev/null` on Darwin). **This is blocked by a fundamental rootless-nested-userns barrier, not a missing config — fully diagnosed on a real Mac (libkrun machine) 2026-06-05:**
  - **Host-side socket is the wrong target.** `podman machine inspect --format '{{.ConnectionInfo.PodmanSocket.Path}}'` yields a *Mac* path (`/var/folders/…/T/podman/podman-machine-default-api.sock`) that doesn't exist inside the VM where the container runs — the mount can't resolve.
  - **VM-internal socket mounts but is inaccessible.** The daemon's native socket is `/run/user/501/podman/podman.sock` (VM user `core`, uid 501, `srw-rw----`). Pointing the stable symlink at that path *does* mount cleanly (virtiofs passes the symlink through; the workspace is shared at the same path so the target resolves in-VM) — but inside the dev container the socket shows as `-?????????` and every access is **EACCES**, for `node` *and* for root in a throwaway. Cause: the socket is owned by `core` at a uid **outside the dev container's keep-id userns map**, and the kernel forbids access to files owned by unmapped uids regardless of mode bits. (The workspace bind mount works only because virtiofs presents *its* files with a mappable owner; the VM-native socket has the raw `core` kuid.)
  - **Idmapped mount is rejected.** Adding `,idmap` (the one configuration-only lever) fails at container creation: `crun: mount_setattr: Operation not permitted` — rootless podman nested in the libkrun VM lacks the privilege.
  - **uid_map for reference** (dev container `/proc/self/uid_map`): `0→1 (×1000)`, `1000→0 (×1)`, `1001→1001 (×999000)`.
  
  **What's left, and why it's deferred.** The only working fixes both add a **standing VM-side service** — against the "keep it simple / compose, don't build / near-zero maintenance" ethos: (a) **TCP** — `podman system service tcp://…` in the VM + `CONTAINER_HOST=tcp://<vm-ip>:port` in the container; technically cleanest (no file-ownership problem) but an unauthenticated API service + open port + VM-IP discovery; (b) **socket proxy** — a sidecar (socat isn't in the FCOS VM) whose own output socket hits the *same* unmapped-owner wall unless it can land on virtiofs (sockets may not bind there) or be owned by a mapped uid. Revisit only if Mac self-testing becomes a felt need; until then macOS contributors build/lint in the dev container and round-trip runtime smoke tests through `konrad-dev` on a Linux host (or the user).

- [ ] **Workspace snapshot / instant undo for agent runs.** Before launching the agent, snapshot the workspace so any destructive operation (mass delete, corrupt rewrite) can be undone with a single command. Implementation sketch: a `konrad snapshot` step (invoked automatically on `konrad run` unless `--no-snapshot`) creates a hidden `.konrad-undo/` directory using the fastest method available — `cp -Rc` (APFS clonefile) on macOS, `cp --reflink=auto` on Linux Btrfs/XFS (zero-space CoW, instant regardless of tree size), falling back to `cp -a` on Ext4/NTFS (still <0.2 s for a typical 100 MB project on NVMe). After the session, `konrad undo` swaps the folders back; `konrad commit-run` discards the snapshot. Advantages over git: survives a deleted `.git` dir, tracks untracked/binary files and `node_modules` without `.gitignore` negotiation, and is faster than `git add && git commit` for large binary assets.
