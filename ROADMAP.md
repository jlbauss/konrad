# Roadmap & Backlog

Tiers gate by release phase: **Tier 1** is the road to beta, **Tier 2** is
beta → 1.0, **Tier 3** is post-1.0. Inside each tier, items cluster by theme.
Each item carries a rough t-shirt size — `S` (hours), `M` (a day or two),
`L` (several days), `XL` (a week+) — a planning guess, not a commitment.
Conventions for what lands here (vs. CHANGELOG or a commit message) are in
[CONTRIBUTING → When to update what](CONTRIBUTING.md#when-to-update-what).

- **Distribution & operations**

  - [ ] `S` **HF token in CI.** Conceptualize the best approach — possibly [trusted publishers](https://huggingface.co/docs/hub/trusted-publishers).
- **Security & sandboxing**
  - [ ] `M` **Container hardening (resource limits, read-only mounts).** Two opt-in `konrad run` flags on `podman_run()`: `--harden` (`--memory` / `--cpus` / `--pids-limit` to cap RAM, CPU, and process count against runaway/fork-bomb scenarios) and `--ro-workspace` (mount `/workspace` read-only + a dedicated writable output volume, so an agent can't modify source it was only asked to read). Network isolation already shipped (egress firewall, 0.8.0). Pairs with the security audit.
  - [ ] `L` **Security audit.** End-to-end review before declaring beta: container isolation, provider-credential handling, MCP tool surface, filesystem-access boundaries.
  - [ ] `S` **Guard `$HOME` as workspace + readable mount-failure error.** Any subcommand run from `$HOME` bind-mounts the entire home dir as `/workspace` (`ws_src="$(pwd -P)"`, [bin/konrad](bin/konrad)) — a data-exposure smell (the sandboxed agent sees all of home), and on macOS the Podman VM can't traverse `~/.Trash` (TCC), so podman dies with a raw `preparing container … for attach: open /Users/<u>/.Trash: operation not permitted` that names neither cause nor fix. Two parts: (a) preflight that refuses (or warns) when `cwd == $HOME`; (b) translate the podman mount failure into a one-line fix ("run konrad from a project directory, not your home folder"). Surfaced 2026-06-17 via `konrad mcp-auth` run from `~`. Folds into the Tier-2 `--doctor` / "name the fix" CLI-polish item if that lands first.

_Work after the public beta. Mostly speculative or aspirational._

Open: structured content trees that flow across pages (`reportlab.platypus`), HTML-to-PDF for richer layouts (Playwright / WeasyPrint), themed/branded output. Likely path: adapt minimax-pdf's CREATE / REFORMAT pipelines (reportlab body + Playwright cover, MIT). Pairs with the Design-skill aesthetic layer.
  - [ ] `L` **DOCX skill — Word authoring + editing.** Mirror the `pdf` / `spreadsheets` skills for `.docx`: read/extract (text, tables, styles, comments, tracked changes), edit (find-replace, tables, paragraph styles, headers/footers), produce from scratch (memos, letters, reports, TOC). Likely path: bundle `python-docx` + a small scaffold, possibly importing MIT-licensed `minimax-docx` the way `spreadsheets` started from `minimax-xlsx`.
  - [ ] `L` **PPTX / presentation skill.** Technical slide authoring.
  - [ ] `M` **`.eml` email skill.** Read and compose `.eml` messages — parse headers / body / attachments and build new messages. Python's stdlib `email` covers most of it; the skill is the scaffold + conventions.
- **Dev Container (second consumption path)**
  - [ ] `XL` **`konrad init-devcontainer`.** Scaffold a `.devcontainer/` into the _user's_ project, so they drive Konrad from VS Code instead of the terminal. Ships as a new subcommand → minor `VERSION` bump + README "Dev Container" section + CHANGELOG; deletes on ship. Not Konrad's own `.devcontainer/` (that configures working _on_ Konrad).
    - **Decided shape:** the published Konrad image _is_ the dev container, sealed on an `--internal` Podman network with the same egress-firewall sidecar the CLI uses — not image-as-toolbox (can't host the firewall — the proxy needs its own netns) and not a socket-mounting re-run of the host CLI (drags in the remote-daemon apparatus for no gain). Used exactly like the CLI (pure agent, no in-container dev work), so it needs no dev egress and the firewall floor stays as tight as the CLI's.
    - **Validated by probe (2026-06-12):** `konrad-entrypoint <anycmd>` composes `baked < org < user` independently of what it execs; an image container on a `--internal` net is fully sealed (DNS fails); the proxy sidecar started _outside_ `podman_run` filters allowed → 200 / denied → 403 — i.e. the firewall assembles by hand exactly as the CLI assembles it.
    - **DRY constraint — reuse, never reimplement:** config compose via `postStartCommand: konrad-entrypoint true`; agent + proxy are the same baked image; extract the network+proxy create/teardown from `fw_setup`/`fw_teardown` into shared helpers used by _both_ the per-run CLI path and a new persistent `konrad fw up/down <id>` the dev container invokes; generate `devcontainer.json` from `bin/konrad`'s existing constants (one source of truth); credentials + model carry via the shared `konrad-secrets` / `-state` / `-cache` volumes. Template is heredoc(s) in `bin/konrad`; image line stamped with the running CLI's `:0.X.Y`.
    - **Open before shipping:** does VS Code provision `vscode-server` host-side (zero container egress — the expected behavior that makes the tight firewall viable) or insist on a container-side fetch (then a single devcontainer-scoped `KONRAD_ALLOWED_HOSTS` entry for just the VS Code host, never the dev set)? Make `konrad fw up` idempotent + self-cleaning; have `--reset` / `--uninstall` sweep stray `konrad-fw-*`.

- **Testing, evaluation & quality**
  - [ ] `L` **Automated test suite.** Today's surface: `smoke-test.sh` (CI-gated, per-arch — binaries, deps, baked content, docling round-trip), `selftest.sh` (end-to-end real `konrad run`, not yet a CI gate — needs a model + credential), `reuse lint` in CI, plus `shellcheck` / `bash -n` as manual gates. Gaps: a _fast pre-commit gate_ that bundles the cheap checks (no ~5.5 min image build), and unit coverage of the logic that has none — [merge-config.js](image/merge-config.js) (the deep-merge / array-replace contract) and `bin/konrad` helpers. Plus a path to run `selftest.sh` in CI (a cheap pinned model or a recorded fixture).
  - [ ] `L` **A/B-test prompts and skills.** Evolve the baked skills and agent prompts empirically: drive konrad end-to-end against a fixed task twice (variant A vs. B) and judge which output is better. Depends on the non-interactive `konrad run` + isolated self-test profile (landed 0.5.0). Substrate: baseline-vs-variant images (or a runtime skill-dir overlay), the same fixed prompt, a pinned cheap/fast model (so cost stays bounded and results aren't confounded by model drift), fixed temperature/seed where allowed, judge-model scoring, isolated `konrad-selftest-*` volumes per run.
  - [ ] `M` **Verify the planning-contract cycle holds.** Empirically test the `task.md → todowrite → quality-assurance` cycle — do the agents actually adhere?
  - [ ] `L` **Quality-assurance subagent.** An independent reviewer subagent that reads `task.md` + the deliverable and renders an out-of-band verdict — the separation-of-concerns the producer's self-check can't provide (same-model self-judging is biased). Open: dispatch model (auto-on-completion vs. agent/user-triggered), conflict-resolution rules vs. the producer, and how it handles language-only deliverables (vision off?).
- **Local models**
  - [ ] `L` **End-to-end shakedown.** Full runs on LM Studio, Ollama, and llama.cpp with the recommended Qwen3.6-class models; fix the remaining rough edges (tool-call parsing, context-overflow handling, model-switch UX).
  - [ ] `M` **Inline model discovery (replaces the dropped plugin).** Replace the removed `opencode-models-discovery` plugin with ~30 lines of host-side bash in `bin/konrad` plus a `jq` step in [entrypoint.sh](image/entrypoint.sh) that writes the discovered providers as a runtime config override. The plugin cost ~3.7 s of Bun startup; doing it natively eliminates that.   - [ ] `M` **GitLab Container Registry mirror.** Make gitlab.git.nrw the single home (repo, installer, images, releases). The build stays GitHub Actions → GHCR (privileged-Podman constraint); add one `skopeo copy --all` at the tail of [build-image.yml](.github/workflows/build-image.yml)'s publish job, mirroring each tag GHCR → `registry.git.nrw/jbauss2/konrad` — done _from_ GitHub Actions, not a GitLab job (which would race the not-yet-published image).   - [ ] `M` **Homebrew tap.** A `homebrew-konrad` repo with a single `Formula/konrad.rb` (downloads a tagged tarball, verifies a `sha256`, depends on `podman`). Defer until (a) actual tagged GitHub releases exist and (b) the GitHub mirror is public.
  - [ ] `S` **Collapse the two-forge setup (standing simplification trigger).** The GitLab-primary / GitHub-mirror split exists for one reason: gitlab.git.nrw's shared runners can't run the privileged Podman build — which radiates into a one-way mirror, a "repo here, CI there" model, and a three-way contributor flow. Revisit whenever the runner situation changes. Supersedes the registry-mirror item above (that keeps GitHub-as-builder; this removes it).
  - [ ] `S` **Improve the image-tag structure.** Revisit the `:0.X.Y` / `:0.X` / `:latest` / dated / `:<sha>` tag scheme (documented in [CONTRIBUTING → Image tags](CONTRIBUTING.md#image-tags)) for clarity.
  - [ ] `M` **`konrad --update` progress UX.** Podman can't reliably render TTY progress bars in macOS remote mode (degrades to raw `Copying blob…`); Linux keeps native progress. ## Blocked (upstream)

_Items that depend on an upstream fix where we've chosen not to ship a workaround. Revisit periodically._

- [ ] `M` **`preserve_thinking` is a no-op through opencode.** Cross-turn reasoning retention does not work on the tested OpenAI-compatible path. Investigated 2026-06-12 (uzk-gpt endpoint / Qwen3.6-27B, full round trip): the endpoint honors `preserve_thinking` only for inline `<think>` in assistant content; it ignores history thinking sent as a separate `reasoning_content` field — the _only_ form opencode emits (it forwards the flag and resends prior reasoning, but never re-inlines `<think>`). So the flag is harmless but does nothing. Decision needed: **(a)** endpoint-side — configure the served chat template / reasoning parser to reconstruct `<think>` from input `reasoning_content` (helps every OpenAI-compatible client); **(b)** opencode-side — re-inline `<think>` into history (needs an upstream change or a plugin); or **(c)** document as a known limitation and drop `preserve_thinking` from the recommended config. Touches the "Local models" shakedown and the A/B-test item's reproducibility needs.
- [ ] `M` **Byte/percent image-pull progress.** A true byte-level or ETA progress bar for `konrad --update` is impossible until Podman populates the byte fields it currently leaves empty — [podman#24887](https://github.com/containers/podman/issues/24887) (open). The skopeo equivalent ([skopeo#658](https://github.com/containers/skopeo/issues/658)) has been open since 2019, and the SSH-bridge / `skopeo → oci-archive` workarounds were rejected as new failure surface against the lightweight ethos. Meanwhile we ship a layer counter + OS-aware append-only progress (see the Tier-2 `--update` progress-UX item). Revisit if #24887 lands — the manifest-fetch + counter scaffolding is most of what a byte consumer needs.
- [ ] `S` **Image build date shows 1970 (`config.created`).** Reproducible builds set `SOURCE_DATE_EPOCH=0`, so `podman images` / `docker inspect .Created` report the epoch. Verified (2026-06) that `config.created` can't be decoupled from the layer-rewrite epoch: dropping `SOURCE_DATE_EPOCH` re-stamps every layer to build time (breaks dedup), and the exporter's `source-date-epoch` attribute overrides `config.created` too — both knobs set one shared value. Only a post-build mutation of the config blob (layers left byte-identical) could give a real date — rejected as not worth the bespoke multi-arch tooling for a cosmetic field. The real build time still lives in `/etc/konrad/build-manifest.json` and the OCI labels. Revisit if the build tooling gains a way to set `config.created` independently of the layer epoch.
