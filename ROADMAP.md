- [ ] **`preserve_thinking` (cross-turn reasoning retention) is a no-op through opencode — a representation mismatch, not a dropped param.** Investigated 2026-06-12 against the `uzk-gpt` endpoint (Qwen3.6-27B), full round trip through `konrad --shell`. opencode + `@ai-sdk/openai-compatible` forwards the model's whole `options` block verbatim to the wire — every sampling param *and* `chat_template_kwargs.preserve_thinking: true` (confirmed by capturing the outgoing request body with `BUN_CONFIG_VERBOSE_FETCH=curl`), and it even resends each prior assistant turn's reasoning as a `reasoning_content` field. But a 3-turn run showed the model could **not** recall a secret it had committed only to an earlier turn's thinking. An isolating probe (direct calls, one variable at a time) pins the cause: historical thinking as **inline `<think>` in assistant content** + `preserve_thinking:true` → recalled ✅; the *same* content as a separate **`reasoning_content` field** (exactly what opencode emits) + `preserve_thinking:true` → **not** recalled ❌; inline `<think>` + `preserve_thinking:false` (control) → not recalled. So the endpoint *does* honor the flag, but only for inline `<think>`; it ignores `reasoning_content` on input, and opencode only ever sends history-thinking as `reasoning_content` (it never inlines `<think>`). **Net: the flag is harmless but does nothing; cross-turn reasoning continuity does not work through opencode on this endpoint.** Decision needed — (a) fix endpoint-side: configure the served chat template / reasoning parser to reconstruct `<think>` from input `reasoning_content` (helps every OpenAI-compatible client, not just konrad); (b) opencode-side: re-inline `<think>` into assistant content for history (needs an upstream change or a plugin); or (c) document as a known limitation and drop `preserve_thinking` from the recommended config. Touches "Local models work flawlessly" (Tier 1) and the A/B-test item's reproducibility needs (Tier 2).

- [ ] **Dev Container as a second consumption path** — a `konrad init-devcontainer` subcommand that scaffolds a `.devcontainer/` into the *user's* project so they can drive Konrad from VS Code's UI instead of the terminal. **Decided shape (after the firewall analysis): the published Konrad image *is* the dev container**, sealed on an `--internal` Podman network with the *same* egress-firewall sidecar the CLI uses — not the simpler "image-as-toolbox" (which can't have the firewall: the proxy needs a daemon and a separate network namespace, and one container is one namespace), and not a socket-mounting container that re-runs the host CLI (drags in the whole remote-daemon/rootful-shim apparatus for no gain). The container is used *exactly* like the CLI — purely the agent, no in-container dev work (no git/curl/npm) — so it needs **no dev egress** and the firewall floor stays as tight as the CLI's, no allow-list widening. **Validated by probe (2026-06-12):** (1) `konrad-entrypoint <anycmd>` composes `baked<org<user` config independently of what it execs; (2) a Konrad-image container on a `podman --internal` net is fully sealed (DNS itself fails); (3) the proxy sidecar started *outside* `podman_run` (on both nets, `KONRAD_ALLOWED_HOSTS`) filters allowed→200 / denied→403 for an agent joined to the internal net only — i.e. the firewall works assembled by hand exactly as the CLI assembles it. **DRY is the design constraint — reuse, never reimplement:** (a) config compose = `postStartCommand: konrad-entrypoint true` (same entrypoint script; runs on every container start regardless of how VS Code keeps the container alive, so opencode launched in the terminal sees the composed config); (b) agent + proxy are the same baked image — no new artifact; (c) **extract the network+proxy create/teardown out of `fw_setup()`/`fw_teardown()` into shared helpers** parametrized by name + lifecycle, called by *both* the per-run CLI path (`$$`-named, EXIT-trap teardown) and a new persistent verb (`konrad fw up/down <id>`, stable name, no trap) that the dev container's `initializeCommand` invokes — the user already has the host CLI, so the dev container delegates firewall orchestration back to it instead of shipping a copy of the logic; (d) **generate `devcontainer.json` from `bin/konrad`'s existing constants** (`SECRETS_VOLUME`/`CACHE_VOLUME`/`STATE_VOLUME`, `KONRAD_CFG_DIR`, the `--userns=keep-id` flag, the `HTTP(S)_PROXY` env, the derived fw-net name) so the mount/env/userns set has one source of truth and a regenerate picks up any change; (e) credentials + model carry over for free by mounting the **same shared `konrad-secrets`/`-state`/`-cache` volumes** the CLI uses. The `.devcontainer/` template is heredoc(s) embedded in `bin/konrad` (the CLI ships standalone — a separate template file wouldn't travel with it). The image line is **stamped with the exact `:0.X.Y` tag from the running CLI's `VERSION`** (invoked as `konrad-dev` → `konrad:local`; bare `dev` → error, or accept `--image-tag`). Refuse-if-exists, `--force` to overwrite. **One open verification before shipping:** whether VS Code provisions `vscode-server` host-side (streamed in over the exec channel, needing zero container egress — the expected Dev Containers behavior, and what makes the tight firewall viable) or insists on a container-side fetch; if the latter, the narrow fallback is a *single* devcontainer-scoped `KONRAD_ALLOWED_HOSTS` entry for just the VS Code download host (passed to `konrad fw up`), never the git/npm dev set. Implementation sketch: a `konrad snapshot` step (invoked automatically on `konrad run` unless `--no-snapshot`) creates a hidden `.konrad-undo/` directory using the fastest method available — `cp -Rc` (APFS clonefile) on macOS, `cp --reflink=auto` on Linux Btrfs/XFS (zero-space CoW, instant regardless of tree size), falling back to `cp -a` on Ext4/NTFS (still <0.2 s for a typical 100 MB project on NVMe).
